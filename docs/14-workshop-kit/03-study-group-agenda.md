# 学习小组议程

这页提供三种共学节奏：

- 90 分钟快速导览
- 半天实操工作坊
- 两周系统共学

你可以按参与者基础和时间选择一种，不需要全部照搬。

## 90 分钟快速导览

适合第一次介绍项目，目标是让参与者理解项目结构和最小运行路径。

| 时间 | 环节 | 内容 | 产出 |
| --- | --- | --- | --- |
| 0 到 10 分钟 | 开场 | 项目定位、学习目标、当前不是生产平台 | 统一预期 |
| 10 到 25 分钟 | 系统地图 | 四个项目分层，gateway 和 inference 边界 | 一张四层图 |
| 25 到 45 分钟 | 本地运行演示 | 文档站、health、model list、chat completion | 最小链路截图或记录 |
| 45 到 60 分钟 | 证据观察 | headers、events、metrics、request timeline | 一条请求证据链 |
| 60 到 75 分钟 | Eval 和 Finetune 快速看 | comparison、leaderboard、manifest、lineage | 两类产物识别 |
| 75 到 90 分钟 | 提问和下一步 | 自测、lab、两周计划 | 每个人选一条后续路径 |

推荐打开页面：

1. [从 0 到 1 学习路径](/00-overview/00-zero-to-one)
2. [四个项目怎么连成系统](/06-projects/06-end-to-end-system-map)
3. [示例输出与证据库](/13-output-gallery/00-overview)
4. [学习者工作簿](/14-workshop-kit/02-learner-workbook)

## 半天实操工作坊

适合小范围带练，目标是让参与者真的跑通并完成一份短复盘。

| 时间 | 环节 | 活动 | 验收 |
| --- | --- | --- | --- |
| 0 到 20 分钟 | 预期对齐 | 说明学习型项目边界和目标 | 每个人能说出今天要完成什么 |
| 20 到 50 分钟 | 环境检查 | `npm install`、`infra-dev-install`、`docs:dev` | 文档站可打开 |
| 50 到 90 分钟 | 请求链路 | 跑 inference 和 gateway 请求 | 能记录 request id |
| 90 到 110 分钟 | 休息与排障 | 处理端口、依赖、环境问题 | 卡点记录 |
| 110 到 150 分钟 | 观测证据 | 查 headers、events、metrics、timeline | 一条请求证据链 |
| 150 到 190 分钟 | Eval 练习 | 跑一次 eval、compare 或查看报告 | 能解释 release recommendation |
| 190 到 220 分钟 | Finetune 练习 | 查看 run manifest、checkpoint、export | 能解释 lineage |
| 220 到 240 分钟 | 复盘展示 | 每个人用 3 分钟讲一条证据 | 一份短复盘 |

推荐要求每位参与者提交：

```text
1. 今天跑通的命令
2. 一个 request id
3. 一个事件或 metrics 证据
4. 一个 eval 或 finetune 产物
5. 一个仍然不确定的问题
```

## 两周系统共学

两周共学可以直接对齐 [两周学习计划](/00-overview/15-two-week-learning-plan)，但建议加上固定的共学产物。

| 天数 | 主题 | 共学产物 |
| --- | --- | --- |
| 第 1 天 | 项目定位和系统地图 | 一张四层系统图 |
| 第 2 天 | 环境和文档站 | 本地启动记录 |
| 第 3 天 | 第一次实操 | 最小链路运行记录 |
| 第 4 天 | LLM serving 基础 | 请求到 token 的流程图 |
| 第 5 天 | inference-service 证据 | 一条 inference 证据链 |
| 第 6 天 | gateway 平台能力 | 一条 gateway 证据链 |
| 第 7 天 | 失败路径和 fallback | 一个失败复盘 |
| 第 8 天 | eval run 和 sample outputs | 一份 eval run 说明 |
| 第 9 天 | comparison 和 release gate | 一个发布判断说明 |
| 第 10 天 | finetune manifest 和 export | 一条训练产物 lineage |
| 第 11 天 | 案例复盘 | 一个案例讲解 |
| 第 12 天 | 证据包整理 | 一份端到端证据包 |
| 第 13 天 | Capstone 准备 | 一份答辩稿 |
| 第 14 天 | 展示和反馈 | 一次完整展示 |

## 每次共学的固定结构

为了让活动稳定，建议每次都用同一套结构：

1. 5 分钟回顾上次产物
2. 10 分钟讲今天的核心问题
3. 25 到 40 分钟阅读和运行
4. 15 分钟整理证据
5. 10 分钟分享一个卡点
6. 5 分钟确定下一步

固定结构比每次临时安排更重要。它会让参与者慢慢形成“读文档、跑命令、看证据、做复盘”的习惯。

## 分组方式

| 分组 | 适合人群 | 任务 |
| --- | --- | --- |
| 新手组 | 第一次接触 AI Infra | 跑通最小链路和工作簿 |
| 系统组 | 有后端或平台经验 | 画系统边界和失败路径 |
| 评测组 | 关心模型质量 | 跑 eval、看 comparison、解释发布建议 |
| 训练组 | 关心微调 | 看 manifest、checkpoint、export lineage |
| 发布组 | 准备维护仓库 | 检查 README、Issue、PR、Pages |

如果参与者水平差异大，不要强行所有人做同一件事。让每组最后用同一个模板汇报即可。

## 结营展示

结营展示建议控制在每人 5 到 8 分钟：

1. 我选择的学习路线
2. 我跑通的链路
3. 我最有价值的一条证据
4. 我遇到的一个失败
5. 我如何定位
6. 我下一步想补的内容

展示时不要只讲“我看了什么”。更好的展示是“我用什么证据证明我理解了什么”。

## 组织者复盘

活动结束后，组织者可以记录：

| 复盘问题 | 用途 |
| --- | --- |
| 哪个页面最多人卡住？ | 改文档结构 |
| 哪个命令最容易失败？ | 改 runbook 或排障 |
| 哪类证据最有帮助？ | 改输出证据库 |
| 哪个问题最常被问？ | 改 FAQ |
| 哪类贡献最自然产生？ | 建 issue 或贡献任务 |

这份复盘可以直接转成后续 GitHub issue。
