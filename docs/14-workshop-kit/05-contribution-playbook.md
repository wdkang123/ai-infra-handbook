# 贡献者协作手册

这页说明如果你把仓库开放到 GitHub，别人应该如何参与贡献。

目标不是让所有贡献都变大，而是让每个贡献都有清楚边界、可验证结果和学习价值。

## 贡献协作的基本原则

这个仓库的贡献不是按“代码行数”衡量，而是按学习价值衡量。一个好的贡献通常满足三件事：

- 让读者更容易理解一个概念、流程或边界
- 让某个命令、输出或排障路径更容易复现
- 让维护者更容易判断改动是否正确

因此，小而清楚的 PR 比巨大但模糊的 PR 更受欢迎。尤其是公开学习站，内容的可读性、证据的可解释性和验证命令的稳定性，本身就是核心质量。

## 从 issue 到 PR 的流转

推荐协作流程：

1. 先用 issue 描述读者卡点或改进目标。
2. 维护者确认范围，补充相关页面、命令和验收标准。
3. 贡献者开小 PR，尽量一次只解决一个问题。
4. PR 描述里写清楚 why、changed files、verification。
5. reviewer 优先检查学习价值、可复现性和公开安全。
6. 合并后如果产生新问题，继续拆成 follow-up issue。

如果你只是修 typo、补一个链接或解释一个字段，可以直接开 PR。
如果你要新增大页面、改导航、调整脚本、引入依赖或重写项目边界，建议先开 issue 讨论。

一个好的 issue 可以这样写：

```text
问题：
读者在阅读 gateway fallback 时不知道应该看哪个输出判断 fallback 生效。

建议：
在输出证据页补一个 fallback 示例，并在 gateway overview 增加链接。

验收：
- docs-quality 通过
- 页面说明 x-upstream-model / failure reason 的含义
- 不引入真实服务 endpoint 或 token
```

## 适合贡献什么

| 类型 | 示例 | 验收重点 |
| --- | --- | --- |
| 文档解释 | 把某个概念讲得更清楚 | 是否降低理解门槛 |
| 命令路径 | 补充某个运行步骤或参数说明 | 是否能被复现 |
| 输出证据 | 补充示例 header、events、JSON、manifest | 是否能帮助读者判断对错 |
| 排障条目 | 补充常见失败和定位方法 | 是否有现象、原因、处理方式 |
| lab 任务 | 新增可执行练习 | 是否有目标、步骤、验收 |
| 案例复盘 | 新增一条跨模块工程故事 | 是否有背景、证据、判断 |
| 测试与脚本 | 增强现有项目质量 | 是否保持 `infra-check` 通过 |

不建议一开始贡献：

- 大范围重写目录结构
- 引入重型依赖
- 把 mock 项目直接替换成生产级实现
- 没有证据的概念堆叠
- 只改样式但不改善学习路径的页面

## 贡献前先判断范围

可以先问自己：

1. 这次改动是文档、代码、脚本、模板还是工作流？
2. 它解决哪个读者卡点？
3. 它会影响哪些页面或项目？
4. 它是否需要更新 README、nav、sidebar 或质量脚本？
5. 它能用哪条命令验证？

如果回答不清楚，先开 issue 描述问题，比直接开大 PR 更好。

## PR 应该怎么写

推荐 PR 描述包含：

```text
## Why
这次改动解决哪个学习卡点？

## What changed
- 改了哪些页面、脚本或项目
- 新增了哪些示例、证据或检查

## Verification
- PYTHON=.venv/bin/python make docs-quality
- 其他相关命令

## Boundaries
- 没有引入真实密钥、真实 endpoint 或生产承诺
- 没有改变不相关目录
```

PR 不需要写得很长，但要让 reviewer 不靠猜。公开仓库里，PR 本身也是后来读者理解项目演进的材料。

## 新增文档页标准

新增页面应该至少包含：

- 一个明确 H1
- 目标读者或使用场景
- 与前后页面的链接
- 关键概念或操作步骤
- 验收方式或复盘问题

如果页面涉及命令，应该说明：

- 在哪个目录运行
- 依赖什么环境
- 成功时应该看到什么
- 失败时先查哪里

如果页面涉及输出证据，应该说明：

- 证据来源
- 证据字段
- 这个字段能证明什么
- 哪些情况不能过度解读

## 修改已有页面的标准

扩写已有页面时，优先补这四类内容：

| 补充方向 | 说明 |
| --- | --- |
| 场景 | 读者什么时候需要这一页 |
| 操作 | 具体看什么文件、跑什么命令、点什么页面 |
| 判断 | 成功、失败、误读分别是什么 |
| 连接 | 它和前后章节、项目、lab、证据之间的关系 |

不要只增加抽象形容词。比如“提高稳定性”不够好，应该写成“当 Pages 404 时，先确认 Source 是否为 GitHub Actions，再确认 `VITEPRESS_BASE` 是否等于项目路径”。

## 新增 Lab 标准

一个合格 lab 至少要有：

| 模块 | 要求 |
| --- | --- |
| 目标 | 读者完成后应该理解什么 |
| 前置条件 | 需要先读哪些页面、跑哪些命令 |
| 操作步骤 | 命令和观察点清楚 |
| 证据要求 | 读者要记录哪些输出 |
| 验收标准 | 怎么判断做完 |
| 延伸问题 | 做完后继续思考什么 |

Lab 不需要复杂，但必须可执行、可观察、可复盘。

## 新增案例标准

案例不是故事作文。一个工程案例应该包含：

1. 背景：发生了什么
2. 症状：读者会看到什么
3. 证据：从哪些接口、文件或报告判断
4. 判断：为什么得出这个结论
5. 修复或下一步：应该怎么处理
6. 复盘：这个案例教会了什么工程原则

可以参考：

- [请求失败排查案例](/11-case-studies/01-request-incident-walkthrough)
- [模型发布判断案例](/11-case-studies/02-model-release-decision-walkthrough)
- [训练产物复现案例](/11-case-studies/03-finetune-to-eval-asset-lineage)

## 新增输出证据标准

新增证据页面或片段时，建议使用这个结构：

```text
场景：
命令或入口：
示例输出：
关键字段：
字段解释：
常见误读：
关联页面：
```

示例输出可以是截取后的关键字段，不需要贴完整日志。重点是让读者知道“应该看哪里”。

## 示例、日志和截图的公开要求

公开仓库可以放示例，但必须是可解释、可脱敏、可复现的示例。

建议：

- 使用 `example.invalid`、`YOUR_API_KEY`、`dev-gateway-key-1` 这类明显占位符
- 截取关键字段，不贴整段带环境信息的日志
- 如果截图包含浏览器、终端路径或账号信息，先裁剪或重新生成
- 示例输出旁边解释字段含义，不让读者靠猜
- 如果输出来自 `.tmp/`，说明它是生成产物，不需要提交

避免：

- 真实 token、cookie、Authorization header
- 真实邮箱、手机号、本机用户名
- 真实业务数据或未授权模型输出
- 无法说明来源的大文件、压缩包、权重和数据库文件

## 代码贡献标准

代码贡献优先遵守现有项目边界：

- inference-service 关注模型服务接口、streaming、events、metrics
- ai-gateway 关注鉴权、路由、限流、fallback、cache、upstream health
- eval-module 关注 run、compare、history、leaderboard、release recommendation
- finetune-demo 关注 dataset、run manifest、checkpoint、export、lineage

新增能力时，至少考虑：

1. 是否有测试
2. 是否有 README 或 docs 更新
3. 是否影响 smoke test
4. 是否保留结构化输出
5. 是否会误导读者以为这是生产级实现

## 提交前检查

最小检查：

```bash
PYTHON=.venv/bin/python make docs-quality
```

文档或配置改动较多时：

```bash
PYTHON=.venv/bin/python make infra-check
```

跨服务或端到端行为改动时：

```bash
PYTHON=.venv/bin/python make infra-smoke
```

格式化：

```bash
PYTHON=.venv/bin/python make infra-format
```

## PR 评审重点

评审时优先看：

| 问题 | 为什么重要 |
| --- | --- |
| 是否真的解决学习卡点 | 避免内容变成堆叠 |
| 是否能被读者复现 | 学习站必须可运行 |
| 是否有证据和验收 | 避免“看起来懂了” |
| 是否破坏导航或链接 | 公开站点不能让读者迷路 |
| 是否把学习型项目说成生产平台 | 保持项目定位诚实 |

## 评审意见怎么回应

收到 review comment 时，推荐这样处理：

- 如果认可，直接改并回复“已处理，验证命令是 ...”
- 如果不确定，先问清楚 reviewer 期望，不急着改一大片
- 如果不同意，解释你担心的副作用，并给出替代方案
- 如果发现问题超出本 PR 范围，拆成 follow-up issue

公开协作里，讨论质量也会影响读者是否愿意参与。目标不是赢一场争论，而是把学习价值、维护成本和验证方式说清楚。

## 维护者分诊节奏

如果仓库开始收到外部 issue，可以用轻量节奏处理：

| 时间 | 动作 |
| --- | --- |
| 24 小时内 | 给问题加标签，确认是否需要更多信息 |
| 3 天内 | 判断是否适合 first-time contributor |
| 1 周内 | 把高价值反馈转成文档、lab、case 或 roadmap task |
| 每月 | 清理重复 issue，更新 FAQ 和公开路线图 |

不需要追求所有问题立刻解决。公开学习项目更重要的是让反馈进入正确的池子：有些变成 FAQ，有些变成 lab，有些变成案例，有些只是记录边界。

## 给 first-time contributor 的任务

适合第一次贡献的任务：

- 补充一个 FAQ 条目
- 为一个 lab 增加“常见失败”说明
- 为一个输出证据页面补充字段解释
- 给某个文档页增加前后链接
- 在 [常见排障手册](/09-reference/04-troubleshooting) 增加一个可复现问题
- 在 [学习者工作簿](/14-workshop-kit/02-learner-workbook) 中补一个记录项

不适合第一次贡献的任务：

- 重构整个 VitePress 配置
- 调整所有目录命名
- 改写 smoke test 主流程
- 大规模替换项目依赖

## 维护者合并前检查

合并前至少确认：

- PR 描述说明了 why
- 新增页面在 sidebar 中出现
- README 关键入口没有过期
- `make docs-quality` 通过
- 涉及代码时对应测试通过
- 涉及端到端链路时 `make infra-smoke` 通过
- 没有真实密钥、私有 endpoint 或敏感日志

如果这些条件不能满足，先缩小 PR 范围。

## 贡献完成后的沉淀

一次贡献合并后，维护者可以顺手检查：

- 是否需要把新问题加入 FAQ
- 是否需要更新公开演示脚本
- 是否产生新的 first public issue
- 是否需要补充 workshop 练习或学习者工作簿
- 是否需要把经验写成案例复盘

这样外部贡献不会只停留在一次 PR，而会逐渐变成更完整的学习材料。
