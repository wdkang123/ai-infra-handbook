# Benchmark 与生产质量不是一回事

## 为什么这一页必须单独讲

因为很多学习者看到 benchmark 分数之后，会自然产生一个很危险的简化：

- 分高 = 线上一定更好

这并不稳。

## benchmark 更擅长回答什么

benchmark 更擅长回答：

- 在固定任务集上，这个模型表现如何
- 在同一口径下，几个模型大概怎么排

它更像“受控条件下的能力对比”。

## 生产质量更像在回答什么

生产质量更像在回答：

- 在你的真实输入里，这个模型是否更适合
- 在你的工作流、提示词、检索、工具调用条件下，结果是不是更稳
- 在质量、成本、延迟之间，这是不是更好的选择

也就是说，生产质量几乎总是更场景化。

## 为什么两者不能直接画等号

因为真实系统通常还受这些因素影响：

- prompt 结构
- RAG / tool 使用方式
- 输出格式要求
- 延迟和成本约束
- 用户偏好

而 benchmark 往往不会完整覆盖这些因素。

## 这对当前仓库意味着什么

当前仓库里的 `eval-module` 之所以强调：

- run
- compare
- history
- bundle

就是因为我们不想把评测理解成“只看一个外部分数”。

更有价值的学习方式是：

1. 看 benchmark 给出的受控结果
2. 再看你自己系统里的 run / compare
3. 再结合 observability 和成本去做判断

## 学习时常见误区

### “排行榜高就直接替换线上模型”

风险很大。  
至少还应该做最小回归和对比。

### “benchmark 没用，真实业务才重要”

也不对。  
benchmark 仍然很有价值，它是稳定对比起点。

### “只要 production feedback 好，就不用 benchmark”

也不稳。  
没有 benchmark，你会缺少一个相对客观、可复现的比较基线。

## 这一章学完应该带走什么

benchmark 很重要，但它更像起点，不是最后裁决。  
真正稳的判断，通常来自 benchmark、run/compare、observability 和场景需求一起看。
