# 四个项目怎么连成系统

## 为什么需要一张系统图

因为单看每个项目页，你容易理解“各自干什么”，  
但未必会自然形成“它们合在一起到底是什么系统”的整体感觉。

这一页就是把四个项目重新放回同一张图里看。

## 最小系统链路是什么

当前仓库最小系统链路可以先理解成：

```text
应用/调用方
  -> ai-gateway
  -> inference-service
  -> eval-module（对结果做评测与比较）
  -> finetune-demo（对训练与导出做资产化）
```

这里面不是所有请求都会顺序经过全部模块，  
但它们合起来，构成了你学习 AI Infra 时最关键的四种能力。

## inference-service 在系统里负责什么

它负责：

- 产出模型响应
- 暴露 health / metrics
- 提供普通响应与 streaming

所以它更像执行层。

## ai-gateway 在系统里负责什么

它负责：

- 统一入口
- 鉴权
- 路由
- 限流
- request id
- upstream health

所以它更像治理层。

## eval-module 在系统里负责什么

它负责：

- 组织 run
- 做 compare
- 落结果 bundle
- 累积 history

所以它更像质量闭环层。

## finetune-demo 在系统里负责什么

它负责：

- 训练 run
- checkpoint
- export
- artifacts / history

所以它更像训练资产层。

## 为什么这四层放在一起很有学习价值

因为它们正好覆盖了 AI Infra 里最常见的四种问题：

- 模型怎么服务
- 请求怎么治理
- 结果怎么判断
- 能力怎么迭代

如果这四层你都建立起基本直觉，后面再看更大系统就会容易很多。

## 当前仓库为什么不用更多项目

不是因为真实世界只有这四层，  
而是因为对学习来说，这四个项目已经足够构成一条完整主线。

如果一开始就再加更多项目，系统地图会变得太散。

## 这一页学完应该带走什么

当前仓库不是四个孤立 demo，而是一张最小 AI Infra 系统图。  
你后面无论读文档还是改代码，都可以先问：我现在动的是执行层、治理层、质量层，还是训练层。

如果你已经理解这张图，下一步可以看 [案例复盘总览](/11-case-studies/00-overview)，用请求失败、模型发布和训练产物复现三个场景把四层真正串起来。
