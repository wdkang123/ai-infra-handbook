# 什么是 AI Infra

## 1. 这是什么

AI Infra 不是单一组件，也不是“把模型跑起来”这么简单。  
如果从工程视角看，它更像一整套围绕模型生命周期建立的基础设施能力，包括模型接入、推理服务、流量治理、缓存、观测、评测、训练微调、成本控制，以及把这些能力落到产品和平台中的工程系统。

对转型学习来说，更实用的定义是：

AI Infra = 围绕模型能力交付、运行、治理和优化的一组工程系统。

## 2. 为什么重要

模型本身再强，如果不能稳定接入、低成本运行、被观测、被评估、被治理，就很难真正支撑业务。

AI Infra 的价值通常体现在几件事上：

- 让模型能力从 demo 变成可上线的服务
- 让推理成本、延迟、吞吐变成可优化对象
- 让接入方式、权限、路由、限流、缓存有统一治理层
- 让效果评估、日志观测、故障排查形成闭环
- 让“用模型”升级成“运营模型系统”

## 3. 核心原理

从学习路径上看，AI Infra 可以拆成四层：

1. 模型运行层：模型如何被加载、调度、推理、并发执行
2. 平台治理层：请求如何被路由、鉴权、限流、缓存、降级和计费
3. 质量闭环层：输出如何被评测、观测、追踪、回放和优化
4. 训练迭代层：数据、微调、评估和线上反馈如何形成正循环

真正难的地方往往不是“知道有哪些组件”，而是理解这些层之间的关系：

- 推理服务决定模型能否稳定跑起来
- AI Gateway 决定能力能否被大规模、规范化使用
- 评测和观测决定系统能否持续优化
- 训练和微调决定能力能否根据场景演进

## 4. 常见方案 / 组件

如果按工程落地来理解，常见组件大致包括：

- 推理服务：`vLLM`、`SGLang`、`Triton`
- 网关与编排：AI Gateway、Router、模型代理层
- 观测与评测：日志、trace、prompt/version 追踪、离线评测、在线回放
- 训练与微调：LoRA、QLoRA、训练数据处理、实验记录
- 支撑能力：缓存、队列、特征开关、配置中心、权限控制、成本看板

重点不在于记住名单，而在于知道每类组件解决什么问题、放在系统的哪一层。

## 5. 关键指标

- 延迟：单次请求多快返回，尤其是首 token 延迟和整体完成时间
- 吞吐：单位时间能处理多少请求或多少 token
- 资源占用：GPU、CPU、显存、内存、带宽的占用情况
- 稳定性：高并发、异常输入、上游波动时是否还能稳定服务
- 成本：单位请求、单位 token、单位吞吐对应的资源成本和运维成本

## 6. 常见误区

- 把 AI Infra 误解为“只会部署模型”
- 只研究模型效果，不研究服务稳定性和成本
- 只看单组件能力，不看系统链路
- 把文档学习和项目实践割裂开
- 一开始就追求大而全，导致迟迟没有可运行成果

## 7. 与项目关系

这一章是整个仓库的总入口，对应后续四条项目线：

- `projects/inference-service/`：理解模型如何稳定提供推理能力
- `projects/ai-gateway/`：理解请求治理、模型接入和平台抽象
- `projects/eval-module/`：理解质量闭环如何建立
- `projects/finetune-demo/`：理解模型能力如何迭代

它的作用不是把所有细节讲完，而是先建立整体地图，避免后续学习碎片化。

你也可以把这四条线暂时记成四个问题：

1. 模型怎么被服务出来
2. 请求怎么被治理起来
3. 结果怎么被判断和比较
4. 能力怎么被继续迭代

后面整套手册其实都在反复回答这四个问题。
