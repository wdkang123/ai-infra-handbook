# 面向分享的学习方式

这套站点后续如果要放到 GitHub 上，最重要的不是“页面数量更多”，而是让第一次来的读者能快速判断：

- 我适不适合学这套内容
- 我应该从哪里开始
- 我每学完一节应该能做出什么
- 我跑完命令后应该看哪些输出证据
- 我能不能用工作簿记录学习过程
- 我怎么验证自己真的理解了
- 如果我想组织共学或贡献内容，应该按什么模板来
- 这个项目当前成熟到什么阶段，哪些边界仍然保留

所以这页不是写给已经熟悉仓库的人，而是写给第一次打开学习网站的人。

## 公开学习项目和个人笔记有什么不同

个人笔记可以只对自己有效：你知道某个缩写是什么意思，也知道某条命令为什么写成这样。公开学习项目不一样，它需要让第一次来的读者也能判断自己站在哪里。

一个公开学习项目至少要回答：

- 这个项目解决什么学习问题。
- 读者需要什么前置基础。
- 第一条路径怎么走。
- 每一阶段完成后应该留下什么证据。
- 哪些内容只是学习型实现，不能直接照搬到生产。
- 遇到问题时应该去哪里查。
- 想贡献时应该怎么开始。

如果这些问题没有写清楚，内容再多也容易让读者迷路。这个网站后续的丰富方向，也应该围绕这些问题持续补强，而不是单纯增加页面数量。

## 这个学习站适合谁

更适合这些读者：

- 已经会写一点 Python 或后端代码，但不清楚 AI Infra 怎么分层
- 会调用大模型 API，但不理解推理服务、网关、评测、微调之间的关系
- 想从“能跑 demo”走到“知道系统为什么这样设计”
- 想用一个小而完整的仓库建立 AI 工程直觉

暂时不适合这些读者：

- 想直接拿生产级 vLLM 集群配置上线
- 想从零学习 Python、HTTP、Docker 或 Linux
- 只想看模型算法论文，不关心工程系统

## 第一次读者的 10 分钟体验

公开分享时，可以假设很多读者只会先给你 10 分钟。他们会快速判断：这个项目是否值得继续看。

这 10 分钟里，最应该让他们看到：

1. 首页能说明项目定位。
2. 从 0 到 1 页面能告诉他第一轮怎么学。
3. 学习路线能告诉他模块顺序。
4. 项目成熟度地图能诚实说明当前边界。
5. 示例输出与证据库能证明项目不是空文档。
6. GitHub 入口能让他知道如何贡献或反馈。

因此，公开学习站的入口页要尽量减少“只有作者懂”的表达。比如“完善 eval 模块”不如写成“让评测结果从单个分数变成可复盘的 run、compare、history 和发布建议”。

## 推荐学习节奏

不要把它当成一本必须顺读的书。更好的方式是：

1. 先跑通最小闭环
2. 再回头补概念
3. 对照输出证据理解结果
4. 再做一个实战 lab
5. 最后对照检查点复盘

也就是说，每一轮都应该包含四件事：

- 读一小段概念
- 跑一条命令
- 看一个产物
- 回答一个为什么

如果只读不跑，很多词会显得抽象。  
如果只跑不读，又容易把系统理解成一堆命令。

### 一轮学习应该如何产出

公开学习不是要求每个人都写长文，但每一轮最好留下一个小产出。

| 学习轮次 | 建议产出 | 可以放到哪里 |
| --- | --- | --- |
| 第一次跑通 | 5 行学习记录 | 学习者工作簿 |
| 学完请求链路 | 一条 request id 证据链 | 输出证据库或个人笔记 |
| 学完 Gateway | 一个失败路径复盘 | case study 草稿 |
| 学完 Eval | 一个发布判断说明 | PR / issue / 复盘模板 |
| 学完 Finetune | 一个产物 lineage 说明 | 端到端证据包 |
| 学完 Capstone | 一段 5 分钟系统讲解 | README、博客或演示稿 |

这些产出不只是为了“记录”。它们会让读者逐步建立表达能力：能把自己做过的事讲清楚，能把证据和判断分开，能说明当前系统还缺什么。

## 每类页面怎么读

### 总览页

总览页回答“这块内容为什么存在”。  
读完总览页，你应该能说清楚这个模块在系统里的位置。

例如：

- inference-service 是执行层
- ai-gateway 是治理层
- eval-module 是质量闭环层
- finetune-demo 是训练资产层

### 概念页

概念页回答“这个词到底在工程里解决什么问题”。  
读概念页时，不要只背定义，而要问：

- 这个概念影响延迟、成本、可用性还是质量
- 它应该出现在系统的哪一层
- 当前仓库用哪个最小实现表达了它

### 项目页

项目页回答“代码里怎么对应”。  
读项目页时，最好一边打开文件一边看，不要只在浏览器里读。

最关键的动作是：

- 找入口文件
- 找请求模型
- 找错误路径
- 找测试覆盖
- 找产物落盘

### 实战 lab

实战 lab 回答“你能不能自己完成一轮工程动作”。  
每个 lab 都有验收标准。只要验收没过，就不要急着进入下一关。

### 案例复盘

案例复盘回答“你能不能把一次工程过程讲清楚”。  
读案例时，不要只看结论，而要看证据顺序：

- 现象是什么
- 第一条证据是什么
- 哪些层被排除
- 最终判断依据是什么
- 还缺哪些生产级能力

### 示例输出与证据库

示例输出页回答“跑完以后看什么”。  
读这类页面时，要重点看：

- 哪个 header、event、JSON 或 manifest 是关键证据
- 这个证据属于哪一层
- 它能证明什么
- 它不能证明什么
- 如何把它写进复盘

### 共学与公开分享套件

共学套件回答“怎么让别人也能跟着学”。  
读这类页面时，要重点看：

- 带练者如何组织一次学习活动
- 学习者如何记录命令、证据和卡点
- 学习小组如何安排议程
- 复盘、PR、issue 和案例如何使用统一模板
- GitHub 首发后如何收集反馈并继续迭代

## 怎么把“读者反馈”变成内容改进

公开之后，读者反馈通常不是直接告诉你“这页缺一个机制解释”。他们更常说：

- 我不知道下一步该看什么。
- 我命令跑完了，但不知道对不对。
- 我看到了 JSON，但不知道哪个字段重要。
- 我感觉案例有结论，但过程不够细。
- 我想贡献，但不知道从哪里开始。

这些反馈可以转成具体改进：

| 反馈 | 对应改进 |
| --- | --- |
| 不知道下一步 | 增加路径提示和下一页链接 |
| 不知道命令是否成功 | 增加预期输出和验收标准 |
| 不知道字段重要性 | 增加字段解释和证据意义 |
| 案例过程不够细 | 增加假设、排除路径和复盘模板 |
| 不知道怎么贡献 | 增加 issue 模板和 good first issue 标准 |

维护公开学习站时，不要只统计访问量。更有价值的是统计读者在哪一步停住、哪类问题重复出现、哪些页面经常被反馈“看不懂”。这些信号会告诉你下一批该丰富哪里。

## 四条公开分享原则

### 1. 先讲系统位置，再讲实现细节

读者最容易卡住的不是某一行代码，而是不知道这行代码属于哪一层。

所以每篇内容都应该尽量先回答：

- 这是执行层、治理层、质量层，还是训练层
- 它和上下游是什么关系
- 它失败时会影响谁

### 2. 每个概念最好能落到一个文件

如果一个概念完全找不到对应文件，读者会很难形成工程直觉。

例如：

- request id 对应 gateway 和 inference 的 header 处理
- fallback 对应 gateway router 和 server
- run bundle 对应 eval 的 result store
- artifact manifest 对应 finetune 的 artifacts 工具

### 3. 每个阶段都要有可验证结果

公开学习项目最怕“看起来学了很多，但不知道学会没有”。

所以这个站点会持续保留：

- 命令
- 输出文件
- 示例输出与证据库
- metrics
- 测试
- smoke
- 检查点
- 阶段自测和 Capstone 答辩稿

这些东西让学习变成可验证的过程。

### 4. 诚实说明边界

公开分享时一定要把学习型边界说清楚。这个项目可以帮助读者理解 AI Infra 主干关系，但它不是生产集群模板。

建议在关键页面反复保留这种表达：

- 当前实现是学习型骨架。
- mock 不代表无价值，它保留了接口和证据边界。
- 生产系统还需要更完整的安全、SLA、调度、审计、监控和回滚。
- 本仓库更适合训练系统判断力，而不是直接复制上线。

诚实说明边界不会削弱项目价值，反而会增强可信度。读者会知道你不是在夸大，而是在帮助他们建立正确迁移路径。

## 对贡献者的期待

如果你后续把它放到 GitHub，建议鼓励贡献者优先做这几类贡献：

- 修正文档中让初学者困惑的表达
- 给概念页补“对应代码位置”
- 给项目页补“失败路径”和“测试路径”
- 给示例输出页补脱敏输出、字段解释和复盘模板
- 给共学套件补更真实的议程、工作簿条目和贡献任务
- 新增可以独立完成的 lab
- 把 mock 实现替换成更真实但仍可教学的实现

不建议一上来就追求“大而全”：

- 不要一下子引入太多基础设施
- 不要为了像生产系统而牺牲可读性
- 不要把每页都写成工具文档

这个仓库最有价值的地方，是它能让读者在一个小系统里看到 AI Infra 的主干关系。

## 贡献者内容质量标准

如果有人想补文档，可以建议他们按下面标准自查：

```text
这页是否说明了目标读者？
这页是否解释了为什么这个主题重要？
这页是否能落到代码、命令、输出或案例？
这页是否说明了常见误区？
这页是否给出完成标准？
这页是否避免过度生产化宣称？
```

如果有人想补代码，可以建议他们按下面标准自查：

```text
这个改动是否保留项目边界？
是否有测试覆盖正常路径和失败路径？
是否更新了对应文档和命令说明？
是否生成或影响输出证据？
是否会引入密钥、本机路径或个人痕迹？
```

这会让贡献者知道，项目欢迎的不只是“大功能”，也欢迎让学习路径更清楚、证据更完整、案例更可复盘的小改进。

## 公开内容的写作语气

公开学习站最好避免两种语气。

第一种是过度营销式：只说“完整、强大、生产级”，但不给证据。
第二种是过度内部化：默认读者知道项目历史、目录结构和作者意图。

更推荐的语气是：

- 先说明问题。
- 再解释为什么重要。
- 然后给最小操作。
- 接着说明应该观察什么。
- 最后说明边界和下一步。

这种写法会让页面更像教程，而不是提纲或宣传页。
