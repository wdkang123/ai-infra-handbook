# 面向分享的学习方式

这套站点后续如果要放到 GitHub 上，最重要的不是“页面数量更多”，而是让第一次来的读者能快速判断：

- 我适不适合学这套内容
- 我应该从哪里开始
- 我每学完一节应该能做出什么
- 我跑完命令后应该看哪些输出证据
- 我能不能用工作簿记录学习过程
- 我怎么验证自己真的理解了
- 如果我想组织共学或贡献内容，应该按什么模板来
- 这个项目当前成熟到什么阶段，哪些边界仍然保留

所以这页不是写给已经熟悉仓库的人，而是写给第一次打开学习网站的人。

## 这个学习站适合谁

更适合这些读者：

- 已经会写一点 Python 或后端代码，但不清楚 AI Infra 怎么分层
- 会调用大模型 API，但不理解推理服务、网关、评测、微调之间的关系
- 想从“能跑 demo”走到“知道系统为什么这样设计”
- 想用一个小而完整的仓库建立 AI 工程直觉

暂时不适合这些读者：

- 想直接拿生产级 vLLM 集群配置上线
- 想从零学习 Python、HTTP、Docker 或 Linux
- 只想看模型算法论文，不关心工程系统

## 推荐学习节奏

不要把它当成一本必须顺读的书。更好的方式是：

1. 先跑通最小闭环
2. 再回头补概念
3. 对照输出证据理解结果
4. 再做一个实战 lab
5. 最后对照检查点复盘

也就是说，每一轮都应该包含四件事：

- 读一小段概念
- 跑一条命令
- 看一个产物
- 回答一个为什么

如果只读不跑，很多词会显得抽象。  
如果只跑不读，又容易把系统理解成一堆命令。

## 每类页面怎么读

### 总览页

总览页回答“这块内容为什么存在”。  
读完总览页，你应该能说清楚这个模块在系统里的位置。

例如：

- inference-service 是执行层
- ai-gateway 是治理层
- eval-module 是质量闭环层
- finetune-demo 是训练资产层

### 概念页

概念页回答“这个词到底在工程里解决什么问题”。  
读概念页时，不要只背定义，而要问：

- 这个概念影响延迟、成本、可用性还是质量
- 它应该出现在系统的哪一层
- 当前仓库用哪个最小实现表达了它

### 项目页

项目页回答“代码里怎么对应”。  
读项目页时，最好一边打开文件一边看，不要只在浏览器里读。

最关键的动作是：

- 找入口文件
- 找请求模型
- 找错误路径
- 找测试覆盖
- 找产物落盘

### 实战 lab

实战 lab 回答“你能不能自己完成一轮工程动作”。  
每个 lab 都有验收标准。只要验收没过，就不要急着进入下一关。

### 案例复盘

案例复盘回答“你能不能把一次工程过程讲清楚”。  
读案例时，不要只看结论，而要看证据顺序：

- 现象是什么
- 第一条证据是什么
- 哪些层被排除
- 最终判断依据是什么
- 还缺哪些生产级能力

### 示例输出与证据库

示例输出页回答“跑完以后看什么”。  
读这类页面时，要重点看：

- 哪个 header、event、JSON 或 manifest 是关键证据
- 这个证据属于哪一层
- 它能证明什么
- 它不能证明什么
- 如何把它写进复盘

### 共学与公开分享套件

共学套件回答“怎么让别人也能跟着学”。  
读这类页面时，要重点看：

- 带练者如何组织一次学习活动
- 学习者如何记录命令、证据和卡点
- 学习小组如何安排议程
- 复盘、PR、issue 和案例如何使用统一模板
- GitHub 首发后如何收集反馈并继续迭代

## 三条公开分享原则

### 1. 先讲系统位置，再讲实现细节

读者最容易卡住的不是某一行代码，而是不知道这行代码属于哪一层。

所以每篇内容都应该尽量先回答：

- 这是执行层、治理层、质量层，还是训练层
- 它和上下游是什么关系
- 它失败时会影响谁

### 2. 每个概念最好能落到一个文件

如果一个概念完全找不到对应文件，读者会很难形成工程直觉。

例如：

- request id 对应 gateway 和 inference 的 header 处理
- fallback 对应 gateway router 和 server
- run bundle 对应 eval 的 result store
- artifact manifest 对应 finetune 的 artifacts 工具

### 3. 每个阶段都要有可验证结果

公开学习项目最怕“看起来学了很多，但不知道学会没有”。

所以这个站点会持续保留：

- 命令
- 输出文件
- 示例输出与证据库
- metrics
- 测试
- smoke
- 检查点
- 阶段自测和 Capstone 答辩稿

这些东西让学习变成可验证的过程。

## 对贡献者的期待

如果你后续把它放到 GitHub，建议鼓励贡献者优先做这几类贡献：

- 修正文档中让初学者困惑的表达
- 给概念页补“对应代码位置”
- 给项目页补“失败路径”和“测试路径”
- 给示例输出页补脱敏输出、字段解释和复盘模板
- 给共学套件补更真实的议程、工作簿条目和贡献任务
- 新增可以独立完成的 lab
- 把 mock 实现替换成更真实但仍可教学的实现

不建议一上来就追求“大而全”：

- 不要一下子引入太多基础设施
- 不要为了像生产系统而牺牲可读性
- 不要把每页都写成工具文档

这个仓库最有价值的地方，是它能让读者在一个小系统里看到 AI Infra 的主干关系。
