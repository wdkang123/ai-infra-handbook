# 常见问题 FAQ

这页回答第一次进入项目时最容易出现的问题。

如果你已经跑完一轮，可以把它当作复盘页；如果你还没开始，可以先快速扫一遍，避免一开始就对项目定位产生误解。

## 这套仓库现在到底是文档项目，还是代码项目

两者都是，但主目标是学习手册优先。

更准确地说，它是一套：

- 以文档为主线
- 以可运行项目为学习支撑
- 以测试、输出证据和 GitHub Pages 为公开发布保障

的学习型工程仓库。

如果只看文档，会少了真实边界；如果只看代码，会少了学习路径。两者合起来才是这个项目的价值。

一个更准确的理解是：这不是“文档站旁边附带代码”，也不是“代码仓库顺手写点文档”。它是把学习路径、可运行边界和验证证据放在同一个仓库里，让读者能从阅读走到实践，再从实践回到复盘。

## 现在这些代码是“真实代码”吗

是学习型真实代码，不是完整生产代码。

也就是说：

- 它们是真的能跑
- 也能形成最小联调闭环
- 有测试和 smoke
- 有 API、CLI、manifest、history 和 evidence
- 但没有一开始就把所有生产复杂度做满

这正是为了让你后面更容易边学边改。

判断“学习型真实代码”的标准不是功能多不多，而是边界是否真实、行为是否可测、输出是否可解释。当前四个项目都保留了这三点。

## 为什么项目里有 mock

因为学习阶段更重要的是先看清系统边界。

Mock 的价值是：

- 降低本地运行门槛
- 避免一开始依赖 GPU、外部 key 或大型模型
- 让接口、错误、metrics、events 和 manifest 可以被稳定测试
- 给后续接真实后端留下清楚边界

Mock 不等于没有工程价值。真正要警惕的是：mock 存在，但边界、证据和迁移路径都说不清。

你可以把 mock 当成一个“可控训练环境”。它让你在没有 GPU、没有 API key、没有真实用户流量的情况下，仍然能学习 request id、streaming、metrics、events、run bundle、manifest 和 lineage。

## 我是不是必须先把所有文档看完才能开始动手

不是。

更推荐的是：

1. 先看总览。
2. 再跑最小手册。
3. 然后做第一次 walkthrough。
4. 再回头读对应章节。

也就是“跑起来”和“看明白”交替进行。

如果你不知道什么时候该停下阅读去动手，可以用一个简单信号：当你已经连续看了三页，但还没有打开终端或输出证据，就该去跑一次命令了。

推荐入口：

- [从 0 到 1 学习路径](/00-overview/00-zero-to-one)
- [最小运行手册](/00-overview/03-runbook)
- [第一次实操演练](/00-overview/04-first-walkthrough)

## 我是不是应该一开始就把代码改得很真实

不建议。

当前更适合的节奏是：

- 先理解结构
- 先跑通主链
- 再做单点小改动
- 之后再逐步把某一层变得更真实

如果一开始就接真实 serving、真实训练、完整 dashboard 和复杂权限，学习成本会很快超过收益。

推荐看：

- [生产迁移路线总览](/12-production-migration/00-overview)

更稳的改法通常是单点替换。比如先替换 inference backend，不同时改 gateway、eval 和训练；或者先给 eval 加一个指标，不同时重做 dashboard。小步验证比大步重构更适合学习项目。

## 如果我跑通了，但还是觉得很多地方是 mock，这正常吗

很正常。

这套仓库本来就是为了支持“先搭骨架，再逐步替换内部实现”的学习路径。

你可以把当前项目理解成：

- 接口是真的
- 测试是真的
- 输出证据是真的
- 系统边界是真的
- 内部执行可以逐步替换

如果你已经能看懂骨架，那就已经走在正确方向上了。

下一步不是否定 mock，而是选择一层逐步变真。比如把 mock inference 接到 OpenAI-compatible backend，或者把 eval 的 mock sample 换成更贴近你关心任务的数据。

## 后面我可以让别的大模型继续帮我细化代码吗

完全可以，而且这是预期用法之一。

更好的方式通常是：

- 先用这套手册建立结构理解
- 再在具体卡点上，用 AI 帮你细化实现或解报错
- 自己运行验证命令
- 自己审查输出证据
- 自己决定是否提交

这样效率会比完全从零问一个模型更高。

建议给 AI 的问题尽量具体：

```text
我正在改 ai-gateway 的 fallback 证据字段。
相关文件是 server.py、router.py、test_proxy.py。
我希望新增字段能出现在 /events/failures。
请先帮我分析应该改哪里，不要直接大改目录结构。
```

这类问题比“帮我完善项目”更容易得到可控结果。

## 我怎么判断自己是不是学明白了一点

最简单的方式不是问“我看了多少页”，而是问：

- 我能不能说清楚四层系统地图
- 我能不能跑通最小闭环
- 我能不能分清 gateway 和 inference-service 的边界
- 我能不能理解 run / compare / history 为什么存在
- 我能不能解释一个 export manifest 的来源
- 我能不能说出一次失败应该先查哪里

如果想更系统一点，就看：

- [学习自测总览](/10-assessments/00-overview)
- [系统地图自测](/10-assessments/01-system-map-check)

如果准备公开分享，再做：

- [Capstone 答辩稿](/10-assessments/04-capstone-defense)
- [公开发布验收 Lab](/07-hands-on-labs/06-public-release-readiness-lab)

另一个很有效的判断方式是“能不能讲给别人听”。如果你能用 5 分钟解释四层系统图，再用 5 分钟讲一个失败案例，说明理解已经开始成形。

## 为什么要用 Node 22

当前项目的文档站和 CI 已经按 Node 22 约束。

本地建议：

```bash
nvm use 22
```

这样本地和 GitHub Actions 更一致，能减少“我这里能跑、CI 不能跑”的问题。

如果你使用 `nvm`，建议进入仓库后先执行：

```bash
nvm use 22
```

再跑 `npm run docs:build` 或本地预览。

## 我应该如何验证改动

取决于你改了什么。

常见选择：

| 改动类型 | 推荐验证 |
| --- | --- |
| 只改文档 | `PYTHON=.venv/bin/python make docs-quality` |
| 准备公开上传 | `PYTHON=.venv/bin/python make public-check` |
| 改项目代码 | `PYTHON=.venv/bin/python make infra-check` |
| 改跨项目链路 | `PYTHON=.venv/bin/python make infra-smoke` |
| 改 npm 依赖 | `npm audit --omit=dev --audit-level=moderate` |

更完整的选择见：

- [验证矩阵](/09-reference/07-validation-matrix)

如果你不确定该跑哪条验证，宁可先跑轻量命令拿反馈，再根据改动范围加重验证。比如文档改动先跑 docs-quality；涉及项目代码再跑 infra-check；涉及跨项目链路再跑 infra-smoke。

## 如果我跑失败了怎么办

先不要急着重装所有东西。

更稳的排查顺序是：

1. 看错误发生在哪条命令。
2. 判断是 Node、Python、端口、依赖、文档构建还是测试失败。
3. 查对应参考页。
4. 如果是请求失败，用 status、headers、request id、events 定位。
5. 如果是文档失败，看 docs-quality 输出的路径和锚点。

推荐入口：

- [常见排障手册](/09-reference/04-troubleshooting)
- [失败症状到证据地图](/13-output-gallery/05-failure-evidence-map)

提问或记录问题时，最好同时留下：

```text
运行的命令：
所在目录：
完整错误摘要：
相关 request id：
是否有 events / metrics：
你已经排除了什么：
```

这样别人或 AI 才能更快帮你定位。

## 什么时候算真正可以进入下一阶段

如果你已经能：

- 跑通最小闭环
- 挑一条主线连续读下去
- 做一个小改动再重跑

## 可以把这个项目直接当生产平台用吗

不能。

当前项目是学习型系统。它可以帮你理解生产系统的边界和迁移路径，但不应该直接拿来承接真实业务流量。

如果要接近生产，需要继续补：

- 真实 serving backend
- 更完整 gateway 策略
- 更强 auth / tenant / quota
- 持久化 observability
- 更真实 eval dashboard
- 真实训练和 artifact store
- 权限、安全、成本、回滚和 SLO

对应入口：

- [生产迁移路线总览](/12-production-migration/00-overview)

它更适合作为生产系统设计前的训练场：先用它理解边界、证据和迁移顺序，再把这些思路带到真实项目里。

## 我准备上传 GitHub，需要注意什么

至少确认：

- 没有 API key、私钥、连接串
- 没有个人本机路径
- 没有未脱敏日志
- public-check 通过
- CI 和 Pages 通过
- README、CONTRIBUTING、SECURITY、LICENSE 都在

对应入口：

- [公开仓库卫生规范](/08-publication/06-public-repo-hygiene)

公开前还建议你站在第一次访问者角度打开首页：他们能不能快速知道项目是什么、从哪里开始、怎么运行、哪里提问题、哪些内容还不是生产级。这些信息越清楚，仓库越容易被认真阅读。

## 这套网站适合写成博客式长文吗

适合，而且当前方向已经在往这个目标推进。

更好的形式不是每页都堆长，而是：

- 核心概念页像教程长文
- 项目页讲代码和边界
- lab 页讲动手路径
- case study 讲复盘故事
- reference 页方便快速查找
- assessment 页帮助验收理解

这样读者既能深入阅读，也能快速定位。

所以后续内容优化不只是“每页变长”，而是让不同类型页面承担不同任务：教程页负责讲透，项目页负责连代码，参考页负责查找，案例页负责训练判断。

## 这一页学完应该带走什么

这套仓库不要求你一开始就把所有东西都学全。

它更像一条渐进路线：先建立系统感，再逐步把实现变真；先跑通，再解释；先自测，再公开分享。
