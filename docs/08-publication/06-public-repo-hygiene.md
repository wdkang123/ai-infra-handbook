# 公开仓库卫生规范

这页用于把仓库从“本地协作项目”收成“可以放心公开分享的学习项目”。

它不追求把所有工作痕迹都藏起来，而是把三类内容分清楚：

- 读者真正需要的学习内容
- 贡献者需要的协作入口
- 维护者本地使用但不适合公开的工作材料

公开仓库卫生的目标不是“看起来干净”，而是让读者能放心 clone、运行、学习和贡献，同时避免密钥、个人信息、私有路径或大文件误入仓库。

## 公开前的三道门

建议把公开前检查分成三道门：

| 门禁 | 目的 | 主要问题 |
| --- | --- | --- |
| 内容门 | 判断是否值得公开 | 这份材料能帮助读者学习或贡献吗 |
| 安全门 | 判断是否安全公开 | 有没有密钥、个人信息、私有路径、真实 endpoint |
| 维护门 | 判断是否能长期维护 | README、命令、导航、检查脚本是否能支撑后续迭代 |

不要只做安全门。一个仓库即使没有密钥，如果 README 入口混乱、命令无法复现、生成产物没有解释，公开后也会让读者迷路。

也不要只做内容门。内容再丰富，如果包含真实 token、账号、日志或私有路径，就不应该公开。

## 公开仓库应该包含什么

公开仓库优先包含：

- 学习文档：`docs/`
- 可运行学习项目：`projects/`
- 自动化脚本和测试：`scripts/`
- GitHub 协作入口：`.github/`
- 脱敏后的任务卡、研究记录和评审记录：`tasks/`
- 脱敏后的 AI 协作提示词和任务模板：`prompts/`
- README、贡献指南、安全说明、发布清单、许可证和更新日志
- 只含占位符的 `.env.example`

这些内容能回答读者最关心的问题：

- 从哪里开始
- 怎么运行
- 怎么验证
- 怎么复盘
- 怎么贡献
- 当前项目边界是什么
- 哪些方向后续可以继续推进

公开仓库不是只给最终用户看，也是在给未来贡献者和维护者看。

## 不应该进入公开提交的内容

默认不提交：

- 真实 `.env`、`.env.*`、`.envrc`、`.npmrc`、`.pypirc`、`.netrc`
- 私钥、证书、云凭证、数据库连接串、带敏感 header 的日志
- `node_modules/`、`.venv/`、缓存、构建产物和临时输出
- 模型权重、checkpoint、adapter、训练日志、大型数据文件和压缩包
- 临时交接文档、个人本机路径、真实账号、真实 endpoint 和未脱敏研究草稿
- 浏览器导出的 cookie、session、HAR 文件和带 token 的 curl 命令

当前仓库通过 `.gitignore` 默认排除这些内容。

任务卡和提示词可以公开，但进入公开仓库前要先完成脱敏，并补充 README 说明，避免读者误以为它们是必须执行的主线。

## 三类材料怎么判断

### 应该公开

满足这些条件，通常适合公开：

- 能帮助读者学习
- 能帮助贡献者理解任务
- 不含真实密钥、账号、endpoint、个人路径
- 能被文档或 README 解释
- 能通过 public-check

### 可以公开但要脱敏

这些内容可以公开，但要处理：

- AI 协作提示词
- 任务拆解记录
- 研究笔记
- 评审记录
- 示例日志
- 示例请求

脱敏时要替换：

- 真实姓名
- 真实邮箱
- 本机用户名
- 私有路径
- 内部 endpoint
- API key
- 业务数据

### 不适合公开

这些内容即使脱敏也通常不适合公开：

- 私钥
- 真实凭证
- 内部合同或账号信息
- 私有数据集
- 未授权模型权重
- 无法解释来源的大型二进制文件
- 带真实用户信息的日志

## 发布前候选文件检查

上传前先看候选入库范围：

```bash
git status --short --ignored
git ls-files --cached --others --exclude-standard
```

重点确认候选文件里没有：

- `base.md`
- `CODEX_IMPLEMENTATION_HANDOFF.md`
- `.tmp/`
- `.venv/`
- `node_modules/`
- `docs/.vitepress/dist/`
- `projects/*/outputs/`
- `projects/*/results/`
- `*.log`
- `*.sqlite`
- `*.zip`
- `*.tar.gz`
- `*.safetensors`

这些目录或文件如果出现在 ignored 区域是正常的；如果出现在待提交区域，就应该先停下来处理。

`tasks/` 和 `prompts/` 可以出现在候选文件中。它们属于公开工作台材料，上传前需要确认其中没有个人路径、真实账号、真实 endpoint、真实 key 或未脱敏日志。

## 发布前安全检查

每次公开上传、发 PR 或打 release 前，运行：

```bash
PYTHON=.venv/bin/python make security-check
```

这个检查只扫描 Git 候选入库文件，覆盖：

- 高置信 API key 和访问令牌形态
- 私钥块
- 数据库、消息队列和缓存连接串
- 本机绝对路径
- 本机用户名、常见个人邮箱、手机号和身份证号
- 日志、数据库、压缩包、模型权重和证书等危险文件类型

它不能替代人的判断，但能挡住最常见的“手滑提交”。

如果准备公开上传，可以直接运行：

```bash
PYTHON=.venv/bin/python make public-check
```

它会把安全检查和主要质量检查连在一起跑。

## 示例值约定

文档和测试里可以使用明显的占位值：

```text
dev-gateway-key-1
example-placeholder-key
YOUR_API_KEY
https://example.invalid/api
req_demo_gateway_1
```

不要使用真实服务里复制出来的 key，也不要把真实 endpoint、tenant id、用户邮箱、请求日志或业务数据当成示例。

好的示例值应该让读者一眼看出：这不是可用凭证，只是教学占位。

## README 和入口检查

公开前，README 至少应该回答：

- 这个项目是什么
- 适合谁
- 怎么本地运行
- 在线站点在哪里
- 文档结构是什么
- 如何验证
- 如何贡献
- 项目是不是生产平台
- 安全问题怎么报告

如果 README 只能服务已经熟悉项目的人，那它还不是一个好的公开入口。

## 文档站检查

公开学习站还要确认：

- 首页能说明项目定位
- 导航能找到核心模块
- 侧边栏没有断链
- 核心章节不是空提纲
- GitHub 入口清楚
- 文档页底部编辑链接可用
- Pages 构建通过

推荐运行：

```bash
PYTHON=.venv/bin/python make docs-quality
```

公开后再确认：

```bash
curl -I https://wdkang123.github.io/ai-infra-handbook/
```

期望看到 `HTTP/2 200`。

## 公开叙事标准

公开仓库的第一视角应该是读者，而不是内部协作者。

因此提交前可以用这组问题自查：

- 新读者能不能从 README 找到起点
- 文档站能不能构建和浏览
- 命令是否能在本机复现
- 示例输出是否能帮助读者判断跑对了
- 当前学习型边界是否说清楚
- 后续贡献入口是否明确
- 失败路径是否有排查入口
- 安全和公开检查是否有说明

如果某份材料主要服务于“我们当时怎么分工、怎么让 AI 生成任务、怎么评审历史草稿”，它更适合留在本地，而不是进入公开仓库。

如果这份材料已经脱敏，并且能帮助读者理解项目如何拆解、研究、评审和迭代，就可以作为公开工作台保留。

## 脱敏的具体做法

脱敏不是把敏感字段随便删掉。好的脱敏应该同时保留学习价值和安全边界。

推荐替换方式：

| 原始类型 | 替换方式 |
| --- | --- |
| 本机路径 | `/path/to/project`、`<repo-root>` |
| 用户名 | `your-name`、`maintainer` |
| 真实邮箱 | `you@example.com` |
| API key | `YOUR_API_KEY`、`example-placeholder-key` |
| 真实 endpoint | `https://example.invalid/api` |
| 请求 ID | `req_demo_gateway_1` |
| 真实业务数据 | `sample input`、`example payload` |

保留字段名和结构，替换字段值。比如示例日志里可以保留 `request_id`、`upstream_model`、`latency_ms`，但值应该是教学占位。

如果一段材料脱敏后已经没有学习价值，就不要公开。不是所有工作痕迹都需要进入仓库。

## 生成产物和大文件策略

`.tmp/`、构建目录、日志、数据库、压缩包、模型权重和训练 checkpoint 默认不进入公开提交。

原因有三点：

- 它们通常可重新生成，不应该污染 Git 历史
- 它们可能包含本机路径、环境信息或运行参数
- 它们会让仓库变重，影响读者 clone 和 review

如果确实需要展示生成结果，优先做成文档页里的“截取示例”：

```text
场景：
命令：
关键输出：
字段解释：
常见误读：
```

如果未来需要托管大模型权重、数据集或二进制 artifact，应该另行设计发布渠道和许可证说明，不要直接塞进学习站仓库。

## 高频本地修改时的发布节奏

当前项目还在快速完善内容和 UI，建议采用批量发布节奏：

1. 本地连续扩写、重构和验证。
2. 每个批次跑 `docs-quality`、`docs:build` 和 `public-check`。
3. 本地预览确认核心入口。
4. 准备一个清楚的提交说明。
5. 到阶段性版本再 push 远端，让 Pages 部署。

这样既能保留高效率，又不会让 GitHub Actions 和公开站点被零碎提交频繁触发。

公开仓库不是不能频繁迭代，而是每次远端更新都应该让读者看到更稳定的一批改进。

## PR 前卫生检查清单

提交 PR 前，建议逐项确认：

```text
[ ] 没有真实 .env 或凭证
[ ] 没有私钥、证书、连接串
[ ] 没有个人路径、真实账号、私有 endpoint
[ ] 没有大型二进制或模型权重
[ ] 示例值都是 placeholder
[ ] README、首页和导航仍然指向正确入口
[ ] 生成产物没有被误加入提交
[ ] docs-quality 通过
[ ] public-check 通过
[ ] CHANGELOG 已记录重要变化
[ ] PR 描述写清验证命令
```

公开项目的卫生不是一次性动作，而是每次提交都要保持的习惯。

## 常见误区

### 误区一：脱敏就是删掉所有工作痕迹

不对。好的任务卡、提示词和评审记录如果脱敏充分，可以帮助读者学习项目如何迭代。

### 误区二：只要 GitHub 没报警就安全

不够。GitHub secret scanning 是最后一层保护，不应该替代本地 `security-check` 和人工判断。

### 误区三：公开前只检查代码，不检查文档

文档里同样可能出现真实路径、endpoint、账号或日志。

### 误区四：构建产物可以顺手提交

不要。VitePress dist、node_modules、venv、outputs、results 都应该由构建或运行命令重新生成。

### 误区五：公开后再慢慢清理

密钥和敏感信息一旦公开，后续处理成本很高。公开前就应该收住。

## 如果发现敏感信息已经公开

如果已经 push 到公开仓库后发现敏感信息，处理顺序应该是：

1. 立刻撤销或轮换相关密钥、token、密码或证书。
2. 删除公开位置里的敏感内容。
3. 判断是否需要重写 Git 历史。
4. 检查 forks、Actions logs、release assets 和 Pages 是否也暴露过。
5. 记录一次复盘：为什么检查没挡住，后续加什么规则。

只在最新 commit 删除敏感内容通常不够，因为 Git 历史里仍然可能保留。
真正的安全动作是先让泄露凭证失效，再处理仓库历史和公开缓存。

这个项目的目标是公开分享，所以宁可在上传前多跑一次 `public-check`，也不要把风险留给发布后补救。
