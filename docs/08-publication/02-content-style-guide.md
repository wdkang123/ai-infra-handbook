# 内容写作规范

这页用于保持学习站长期可维护。

当页面越来越多时，最大的问题不是“内容不够”，而是：

- 新读者不知道先看什么
- 页面风格不一致
- 概念和代码脱节
- 命令输出没有解释
- 改了代码但文档没同步
- 共学材料没有明确对象和产物
- 新增页面没有验收标准

## 写作质量分层

为了避免页面看起来像目录，可以把内容质量分成四档：

| 档位 | 页面表现 | 读者读完后的状态 |
| --- | --- | --- |
| 提纲 | 只有名词、要点和链接 | 知道主题存在，但不知道怎么用 |
| 说明 | 能解释概念和边界 | 大致理解，但还不能独立复现 |
| 教程 | 有场景、机制、步骤、证据、误区和仓库映射 | 能按步骤操作，并能解释为什么 |
| 可复盘 | 有成功路径、失败路径、判断标准、产物样例和下一步行动 | 能把结果讲给别人听，也能转成 issue、PR 或 release evidence |

公开分享时，核心页面至少应该达到“教程”档。
Lab、案例、证据库、发布手册这类页面最好接近“可复盘”档，因为它们承担的不只是解释概念，还要帮助读者形成工程判断。

## 页面基本结构

新增学习页面时，优先使用这个结构：

```md
# 标题

## 这页解决什么问题

## 你需要先知道什么

## 核心概念

## 当前仓库怎么表达

## 应该观察什么

## 常见误区

## 学完应该能回答什么
```

不一定每页都完整照抄，但要尽量让读者知道：

- 为什么读这页
- 读完能做什么
- 和代码哪里对应

## 深度教程页标准

公开分享时，核心章节不能只停留在“名词解释 + 要点列表”。
读者真正需要的是：看完这一页以后，能把概念放进一个真实工程场景里，知道它为什么出现、怎么工作、如何观察、哪里容易误判，以及当前仓库用什么最小实现表达了这个概念。

因此，核心概念页和教程页应该尽量满足下面这些标准。

### 一页至少回答六类问题

一篇有学习价值的教程页，至少要覆盖：

| 问题 | 说明 |
| --- | --- |
| 为什么重要 | 这个概念解决什么工程问题，忽略它会造成什么后果 |
| 放在哪一层 | 它属于模型、推理运行时、网关、评测、训练、发布还是维护流程 |
| 机制是什么 | 用读者能跟上的方式解释内部过程，不只给定义 |
| 如何观察 | 通过哪些日志、header、metrics、artifact、报告或命令输出判断它是否发生 |
| 取舍是什么 | 它通常会影响延迟、吞吐、成本、稳定性、安全性还是可维护性 |
| 仓库怎么表达 | 当前学习型实现在哪些文件、API、CLI、lab 或测试里表达了这个概念 |

如果一页只能回答“它是什么”，通常还只是提纲；如果能回答“为什么系统要这样设计，以及我怎么验证”，才更接近教程。

### 推荐篇幅和密度

不同页面允许有不同长度，不需要所有页面都写成博客长文：

| 页面类型 | 推荐深度 |
| --- | --- |
| Overview / index | 可以短，但要清楚说明本章地图、学习顺序和下一步 |
| 核心概念 / 教程页 | 通常建议 1500-4000 个中文字符，包含场景、机制、取舍、误区和仓库映射 |
| Lab / hands-on | 不只列命令，要写目标、前置条件、步骤、预期输出、失败路径、验收标准和复盘问题 |
| 案例复盘 | 应该有时间线、症状、假设、证据、排除过程、根因、改进和可迁移经验 |
| 参考表 / checklist | 可以以表格为主，但要说明何时使用、怎么判断、不能证明什么 |
| 生成产物页 | 可以保持结构化，但要说明生成来源、适用场景和人工复核点 |

短页不是问题，问题是核心页没有带读者完成一次理解闭环。
如果页面标题承诺了一个重要主题，例如 KV Cache、Gateway rate limit、Eval leaderboard、LoRA，那么它就应该像一篇可独立阅读的教程，而不是章节目录。

### 核心页建议结构

核心页不必机械套模板，但建议至少包含这些部分：

```md
# 标题

## 这页解决什么问题

## 一个具体场景

## 核心机制

## 工程取舍

## 当前仓库怎么表达

## 如何验证或观察

## 常见误区

## 学完应该能回答

## 继续阅读
```

其中“一个具体场景”很重要。
例如：

- 讲 KV Cache 时，不要只说缓存 key/value；要解释长上下文请求为什么 prefill 慢，decode 为什么逐 token 生成，以及 serving 系统为什么要管理显存。
- 讲 AI Gateway 时，不要只列鉴权、路由、限流；要解释平台为什么不应该让业务方直接写死模型供应商，以及 401、429、502 分别代表什么边界。
- 讲 Eval 时，不要只说 benchmark 和 leaderboard；要解释为什么发布前需要固定数据集、参数、judge 策略和历史对照。
- 讲 LoRA/QLoRA 时，不要只说参数高效微调；要解释为什么 adapter 是工程资产，以及为什么微调必须接评测闭环。

### 判断页面是否还太薄

可以用这些问题自查：

- 这页有没有一个真实问题，而不只是定义集合？
- 有没有解释机制，而不是只给结论？
- 有没有说清楚成功路径和失败路径？
- 有没有告诉读者在当前仓库哪里能看到对应实现？
- 有没有至少一个表格、流程、例子、误区或复盘问题帮助读者形成结构？
- 读者看完后，能不能向别人讲清楚“为什么要这样做”？

如果答案大多是否定，这页应该继续扩写。

### 改写示例：从提纲到教程

薄页面常见写法是：

```text
KV Cache 可以提升推理速度。它缓存历史 token 的 key/value，减少重复计算。
```

这句话没有错，但读者很难继续往下走。
更好的教程式写法应该补齐上下文：

```text
当模型生成第 101 个 token 时，前 100 个 token 已经产生过 attention 的 key/value。
如果每次 decode 都重新计算这些历史状态，长上下文请求会浪费大量显存带宽和计算。
KV Cache 的作用，是把历史 token 的 key/value 保留下来，让后续 decode 只关注新增 token。

工程上它会带来新的取舍：缓存越多，decode 越快，但显存占用越高；batching 越激进，吞吐可能提升，但单个请求的排队和缓存管理更复杂。
在本仓库里，你可以先通过 serving 的 streaming、usage、metrics 和 request timeline 理解“生成过程是逐 token 发生的”，再回到推理章节理解真实 serving 系统为什么要管理 KV Cache。
```

改写的关键不是把句子拉长，而是补上“场景 -> 机制 -> 取舍 -> 观察方式 -> 仓库入口”。
这样的页面才更像博客式教程，而不是概念卡片。

### 允许短，但要短得有用

有些页面本来就适合短，例如章节 overview、导航页、清单页、入口页。
这类页面不需要强行写成长文，但至少要承担清楚的导航责任：

- 告诉读者本章要解决什么问题
- 告诉读者先读哪几页
- 告诉读者哪几页适合动手
- 告诉读者学完后能进入哪个案例、lab 或项目

换句话说，短页可以是路标；核心页必须是教程。

## Lab 页面结构

新增 hands-on lab 时，使用这个结构：

```md
# Lab 名称

## 学习目标

## 前置知识

## 代码入口

## 操作步骤

## 关键观察点

## 扩展任务

## 验收标准
```

一个 lab 最好满足：

- 能独立完成
- 有明确命令
- 有明确观察点
- 有明确验收方式
- 不要求读者先懂完整系统

## 示例输出页面结构

新增输出证据页面时，优先使用这个结构：

```md
# 证据主题

## 这一页看什么

## 关键命令

## 应该关注的字段

## 这些字段能证明什么

## 这些字段不能证明什么

## 常见误读

## 复盘模板
```

输出证据页不要只贴完整 JSON。更好的做法是：

- 摘出关键字段
- 解释字段属于哪一层
- 写清楚它能证明什么
- 写清楚它不能证明什么
- 链接到对应 lab、案例和参考页

## 视觉资产与截图规范

网站可以使用截图、架构图、封面图或生成式图片，但它们必须服务于学习，而不是只做装饰。

推荐使用视觉资产的场景：

| 场景 | 适合的资产 | 目的 |
| --- | --- | --- |
| 首页或分享卡片 | 项目封面图、社交分享图 | 让读者第一眼知道这是 AI Infra 学习站 |
| Lab 页面 | 终端输出截图、浏览器截图、证据链示意图 | 帮读者判断自己跑出来的结果是否接近预期 |
| 案例复盘 | 时间线、排障路径图 | 说明证据如何一步步收敛到原因 |
| 架构解释 | 请求链路图、分层图 | 帮读者区分 serving、gateway、eval、training 的责任 |

使用生成式图片时要注意：

- 不要生成真实公司、产品或开源项目 logo，避免暗示官方背书。
- 不要用图片替代证据；截图、JSON、manifest、report 仍然要能被文字解释。
- 不要把视觉风格做得像营销页，学习站更适合清晰、克制、可阅读。
- 生成图片如果用于 `og:image` 或封面，要保存在 `docs/public/` 下，并在发布前检查移动端裁切。
- 所有图片都要有明确用途：解释一个概念、展示一个状态、帮助公开分享，或提升 GitHub/社交入口识别度。

如果后续增加分享图，建议补齐：

```text
docs/public/og-image.png
docs/.vitepress/config.mts 中的 og:image 和 twitter:image
README 中的项目截图或封面
```

## 共学页面结构

新增带练、工作簿、议程或贡献协作页面时，优先使用这个结构：

```md
# 共学主题

## 这页给谁用

## 使用场景

## 准备材料

## 操作步骤或活动节奏

## 参与者应该产出什么

## 常见卡点

## 后续如何反馈或贡献
```

共学页面不要只写抽象建议。更好的做法是：

- 给出具体时长或流程
- 给出参与者要填写的模板
- 给出组织者要收集的反馈
- 链接到 [共学与公开分享套件](/14-workshop-kit/00-overview)

## 代码引用规范

引用本地文件时，尽量写完整相对路径，例如：

```text
projects/ai-gateway/src/ai_gateway/server.py
```

不要只写 `server.py`，因为仓库里多个项目都有入口文件。

## 命令规范

命令尽量从仓库根目录或项目目录开始说明。

推荐写法：

```bash
cd /path/to/ai-infra
PYTHON=.venv/bin/python make infra-check
```

如果命令需要项目目录，也要明确：

```bash
cd /path/to/ai-infra/projects/eval-module
PYTHONPATH=src ../../.venv/bin/python -m eval_module.main run ...
```

## 概念解释规范

解释概念时，优先回答工程问题：

- 它解决什么问题
- 它在哪一层出现
- 它影响什么指标或产物
- 当前仓库哪个文件能看到它

例如不要只写：

```text
TTFT 是 Time To First Token。
```

更好的写法是：

```text
TTFT 是用户看到第一个 token 前的等待时间。它通常受 prefill、排队、batching 和上游健康状态影响。
```

## 错误路径必须写清楚

AI Infra 学习里，失败路径往往比成功路径更能体现系统设计。

涉及 API、CLI、manifest、cache、fallback 的页面，尽量说明：

- 成功时是什么行为
- 失败时是什么行为
- 测试覆盖在哪里
- metrics 或 artifact 如何反映失败
- 如果有示例输出，应该链接到 [示例输出与证据库](/13-output-gallery/00-overview)

## 不要把学习型实现伪装成生产级系统

当前项目是学习型系统。

写文档时要保持边界清楚：

- 可以说“这里表达了生产系统中的某个概念”
- 不要说“这就是生产级实现”

推荐表达：

```text
当前实现用于建立概念边界，还不是生产级缓存系统。
```

## 更新导航

新增页面后，请检查：

- `docs/.vitepress/config.mts`
- 相关 overview 页面
- 如果是输出证据，更新 [示例输出与证据库总览](/13-output-gallery/00-overview)
- 如果是共学材料，更新 [共学与公开分享套件](/14-workshop-kit/00-overview)
- README 是否需要新增入口

如果页面没有入口，读者很难发现它。

## PR 评审检查清单

文档 PR 不只检查有没有错字，也要检查学习闭环：

| 检查项 | 通过标准 |
| --- | --- |
| 目标清楚 | 页面开头能说明读者为什么要看这页 |
| 场景具体 | 至少有一个真实工程问题或操作场景 |
| 机制可解释 | 不只给定义，还解释为什么会这样设计 |
| 代码有入口 | 能指到当前仓库的文件、CLI、API、lab 或产物 |
| 证据可验证 | 有命令、输出字段、artifact、report 或检查方式 |
| 失败路径存在 | 至少说明一个常见错误、误判或卡点 |
| 公开边界清楚 | 没有把学习型实现写成生产级承诺 |
| 安全卫生合格 | 没有密钥、私有 endpoint、个人路径或敏感日志 |

如果一个 PR 只增加抽象段落，没有代码入口、证据或验收标准，最好继续要求补充。

## 提交前检查

文档改动后至少运行：

```bash
nvm use
npm run docs:build
```

并运行文档质量检查：

```bash
PYTHON=.venv/bin/python make docs-quality
```

如果涉及代码或命令，也运行：

```bash
PYTHON=.venv/bin/python make infra-check
```
