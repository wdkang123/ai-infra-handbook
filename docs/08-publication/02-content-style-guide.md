# 内容写作规范

这页用于保持学习站长期可维护。

当页面越来越多时，最大的问题不是“内容不够”，而是：

- 新读者不知道先看什么
- 页面风格不一致
- 概念和代码脱节
- 命令输出没有解释
- 改了代码但文档没同步
- 共学材料没有明确对象和产物
- 新增页面没有验收标准

## 页面基本结构

新增学习页面时，优先使用这个结构：

```md
# 标题

## 这页解决什么问题

## 你需要先知道什么

## 核心概念

## 当前仓库怎么表达

## 应该观察什么

## 常见误区

## 学完应该能回答什么
```

不一定每页都完整照抄，但要尽量让读者知道：

- 为什么读这页
- 读完能做什么
- 和代码哪里对应

## Lab 页面结构

新增 hands-on lab 时，使用这个结构：

```md
# Lab 名称

## 学习目标

## 前置知识

## 代码入口

## 操作步骤

## 关键观察点

## 扩展任务

## 验收标准
```

一个 lab 最好满足：

- 能独立完成
- 有明确命令
- 有明确观察点
- 有明确验收方式
- 不要求读者先懂完整系统

## 示例输出页面结构

新增输出证据页面时，优先使用这个结构：

```md
# 证据主题

## 这一页看什么

## 关键命令

## 应该关注的字段

## 这些字段能证明什么

## 这些字段不能证明什么

## 常见误读

## 复盘模板
```

输出证据页不要只贴完整 JSON。更好的做法是：

- 摘出关键字段
- 解释字段属于哪一层
- 写清楚它能证明什么
- 写清楚它不能证明什么
- 链接到对应 lab、案例和参考页

## 共学页面结构

新增带练、工作簿、议程或贡献协作页面时，优先使用这个结构：

```md
# 共学主题

## 这页给谁用

## 使用场景

## 准备材料

## 操作步骤或活动节奏

## 参与者应该产出什么

## 常见卡点

## 后续如何反馈或贡献
```

共学页面不要只写抽象建议。更好的做法是：

- 给出具体时长或流程
- 给出参与者要填写的模板
- 给出组织者要收集的反馈
- 链接到 [共学与公开分享套件](/14-workshop-kit/00-overview)

## 代码引用规范

引用本地文件时，尽量写完整相对路径，例如：

```text
projects/ai-gateway/src/ai_gateway/server.py
```

不要只写 `server.py`，因为仓库里多个项目都有入口文件。

## 命令规范

命令尽量从仓库根目录或项目目录开始说明。

推荐写法：

```bash
cd /path/to/ai-infra
PYTHON=.venv/bin/python make infra-check
```

如果命令需要项目目录，也要明确：

```bash
cd /path/to/ai-infra/projects/eval-module
PYTHONPATH=src ../../.venv/bin/python -m eval_module.main run ...
```

## 概念解释规范

解释概念时，优先回答工程问题：

- 它解决什么问题
- 它在哪一层出现
- 它影响什么指标或产物
- 当前仓库哪个文件能看到它

例如不要只写：

```text
TTFT 是 Time To First Token。
```

更好的写法是：

```text
TTFT 是用户看到第一个 token 前的等待时间。它通常受 prefill、排队、batching 和上游健康状态影响。
```

## 错误路径必须写清楚

AI Infra 学习里，失败路径往往比成功路径更能体现系统设计。

涉及 API、CLI、manifest、cache、fallback 的页面，尽量说明：

- 成功时是什么行为
- 失败时是什么行为
- 测试覆盖在哪里
- metrics 或 artifact 如何反映失败
- 如果有示例输出，应该链接到 [示例输出与证据库](/13-output-gallery/00-overview)

## 不要把学习型实现伪装成生产级系统

当前项目是学习型系统。

写文档时要保持边界清楚：

- 可以说“这里表达了生产系统中的某个概念”
- 不要说“这就是生产级实现”

推荐表达：

```text
当前实现用于建立概念边界，还不是生产级缓存系统。
```

## 更新导航

新增页面后，请检查：

- `docs/.vitepress/config.mts`
- 相关 overview 页面
- 如果是输出证据，更新 [示例输出与证据库总览](/13-output-gallery/00-overview)
- 如果是共学材料，更新 [共学与公开分享套件](/14-workshop-kit/00-overview)
- README 是否需要新增入口

如果页面没有入口，读者很难发现它。

## 提交前检查

文档改动后至少运行：

```bash
nvm use
npm run docs:build
```

并运行文档质量检查：

```bash
PYTHON=.venv/bin/python make docs-quality
```

如果涉及代码或命令，也运行：

```bash
PYTHON=.venv/bin/python make infra-check
```
