# 从请求到首个 Token

## 为什么这一页很重要

很多工程词之所以会显得抽象，是因为你还没有把“一次请求在系统里到底经历了什么”串起来。

如果这条链路不清楚，后面你看到：

- prefill
- decode
- TTFT
- streaming
- request id
- gateway proxy

就容易像在看一堆彼此无关的术语。

## 一条请求最粗的路径是什么

先用最粗的方式看：

1. 应用发起请求
2. gateway 接住并治理
3. inference-service 开始执行
4. 模型处理 prompt
5. 产出首个 token
6. 后续 token 持续生成
7. 结果返回给调用方

这条路径看起来简单，但你后面几乎所有系统问题都会落在这几步中的某一段。

## 在首个 token 之前发生了什么

这是学习里非常关键的一段。  
很多人会觉得“模型不是收到 prompt 就开始吐 token 吗”，但工程上真正影响首个 token 的，是前面那一整段准备过程。

大致包括：

- 请求进入服务
- 鉴权与路由
- prompt 组装完成
- 模型做 prefill
- cache 命中或未命中

所以 TTFT 从来不只是“模型生成速度”的问题。  
它常常是整条链路共同作用的结果。

## 为什么首个 token 比后续 token 更敏感

因为首个 token 之前，需要做的准备最多。

后续 token 阶段通常更像持续 decode；  
但在首个 token 之前，系统往往已经经历了：

- 网络进入
- 入口治理
- prompt 处理
- prefill
- 初次调度

这也是为什么大家会把 TTFT 单独拿出来看，而不是只看总时长。

## streaming 为什么会改变你的观察方式

如果没有 streaming，你看到的常常只是“多久返回了一整个结果”。  
但有了 streaming 后，你开始能把这件事拆开：

- 首个 token 什么时候来
- 后面 token 是快还是慢
- 流有没有中断

也就是说，streaming 不只是交互体验，它也让系统行为更容易被观察。

## 在当前仓库里怎么对应

当前仓库就是故意把这条链拆成几个清楚的层次给你看：

- `ai-gateway` 负责入口治理
- `inference-service` 负责执行与结果产出
- `x-request-id` 帮你把同一条请求串起来
- `/metrics` 帮你看整体状态
- streaming 帮你观察从首个 token 到后续 token 的阶段差异

所以这套仓库并不是在教你一个孤立 API，而是在教你如何看一条请求的完整生命周期。

## 这一页学完应该带走什么

你最应该带走的是这个直觉：

从请求到首个 token，不是模型内部的一瞬间，而是一整条系统路径。  
后面几乎所有 AI Infra 问题，都能在这条路径上找到位置。
