# 模型、Token、Context

## 模型是什么

在工程语境里，模型可以先粗暴理解成“一个根据上下文预测下一个 token 的函数”。  
这个理解不完整，但足够支撑你读后面的系统设计。

它意味着两件事：

- 模型不是直接理解整段自然语言，而是在处理 token 序列
- 模型每次输出都依赖于前面已经看到的上下文

## Token 是什么

token 不是“一个字”也不是“一个单词”，而是模型分词器切出来的最小处理单元。  
这件事很重要，因为后面很多工程问题都不是按“字符数”算，而是按 token 数算：

- 请求成本
- 上下文长度
- 推理延迟
- 吞吐能力

所以你后面看到 `prompt_tokens`、`completion_tokens`、`total_tokens`，其实就是在看一次请求到底消耗了多少模型处理量。

## Context 是什么

context 可以理解成“模型这次推理时能够看到的全部输入窗口”。  
它通常包括：

- system prompt
- user / assistant 历史消息
- 当前这条新输入

context 不是越大越好，因为它会直接带来：

- 更高的 prefill 成本
- 更高的显存压力
- 更长的等待时间

所以 context 同时是能力边界和系统成本边界。

## 学这一节最重要的直觉

后面你看到任何服务接口、网关指标、评测结果，最好都先在脑子里问自己三个问题：

1. 这次到底处理了多少 token？
2. 这次上下文到底有多长？
3. 这个设计是在优化能力，还是在优化成本？

如果这三个问题能逐渐形成习惯，后面的 AI Infra 内容会顺很多。
