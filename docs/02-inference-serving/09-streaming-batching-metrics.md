# Streaming、Batching、Metrics

## 为什么这三个词应该一起看

因为它们分别对应三种很核心的系统视角：

- streaming：用户怎么感知结果
- batching：系统怎么组织执行
- metrics：我们怎么观察整体状态

如果只学其中一个，你对推理服务的理解会很容易偏掉。

## Streaming 在回答什么

streaming 回答的是：

**结果能不能一边生成一边交给调用方。**

它首先影响的是体验，但很快也会影响系统设计：

- 首个 token 什么时候出现
- 中途失败怎么表达
- gateway 能不能继续透传事件流

所以 streaming 是连接“用户体验”和“服务边界”的一个点。

## Batching 在回答什么

batching 回答的是：

**多个请求怎么一起组织执行，才不至于把硬件白白浪费掉。**

这件事对吞吐极其关键。  
但学习时最重要的不是先背复杂策略，而是意识到：

- 在线推理不是单请求世界
- batching 是服务系统行为，不只是模型参数

## Metrics 在回答什么

metrics 回答的是：

**最近这套服务整体处于什么状态。**

比如：

- 请求数
- 成功率
- 失败率
- 正在执行的请求数
- token 统计

这层会帮你把“某一条请求发生了什么”扩展成“最近整个服务怎么样”。

## 为什么这三者要一起学

因为它们正好构成了推理服务最关键的三个观察角度：

### 用户视角

我是不是很快看到首个 token，streaming 是否顺畅。

### 系统执行视角

请求是不是被合理地 batch 起来，资源有没有被有效利用。

### 运维视角

最近错误率、请求数、运行态指标是不是正常。

这三种视角合在一起，你才真正是在看“服务”，而不是只看“模型”。

## 在当前仓库里怎么对应

当前仓库已经把这三条线都做成了最小可观察形态：

- `inference-service` 支持普通响应和最小 streaming
- `ai-gateway` 已能透传 streaming
- `/metrics` 已有真实运行态计数
- smoke 会验证 streaming、health、metrics、request id

所以你现在可以直接把文档里的概念对回代码和运行结果，而不是停在抽象层。

## 学习时常见误区

### “streaming 只是更好看”

不止。  
它会改变你如何观察延迟，也会改变代理层和错误处理方式。

### “batching 只是大模型框架内部的事”

也不对。  
它虽然更靠执行层，但它直接影响服务吞吐和整体行为。

### “metrics 只是部署上线后才需要”

其实越早看越好。  
因为它会帮你把服务从“能跑起来”提升到“能被理解”。

## 这一章学完应该带走什么

streaming、batching、metrics 分别站在体验、执行、观察三个方向。  
把这三者同时放进脑子里，你对推理服务的理解才会真正立体起来。
