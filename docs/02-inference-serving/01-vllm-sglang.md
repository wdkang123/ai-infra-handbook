# vLLM 与 SGLang

## 为什么这两者经常一起出现

因为它们都站在“LLM 推理服务框架”这个位置上。  
也就是说，它们都在解决：

- 请求如何进入服务
- 模型如何高效生成
- batching / cache / streaming 如何组织

所以大家常常会把它们放在一起比较。

## vLLM 更适合先理解成什么

先把 vLLM 理解成“围绕高效 LLM serving 做得很系统的一条主线”。  
它之所以常被提起，是因为它让很多原本很重的推理问题，变成了更可落地的服务能力。

对学习来说，vLLM 最重要的价值不是记住所有特性，而是理解：

- serving 框架不是简单包一层 HTTP
- 真正难的是如何把请求、缓存、生成、吞吐协调起来

## SGLang 更适合先理解成什么

SGLang 一般更容易让人联想到“更强调生成流程编排、服务执行和系统层协同”的路线。  
你可以先把它看成另一种对“高性能生成服务”问题的回答。

学习时不必急着判断“谁绝对更好”，更值得观察的是：

- 两者都在优化 prefill / decode / cache
- 但它们强调的执行组织和系统表达方式不完全一样

## 学习时最值得关注什么

先不要陷入功能清单，优先看三件事：

1. streaming 怎么做
2. cache 怎么管
3. 批处理和吞吐怎么取舍

如果这三件事能看明白，你就已经抓住了大部分核心。
