# 服务选型与取舍

## 先别急着问“哪个最好”

推理服务选型很少是一个绝对答案，通常都要先看你更在意什么：

- 单请求体验
- 总体吞吐
- 硬件生态
- 部署复杂度
- 可维护性

## 常见取舍轴

### 1. 体验 vs 吞吐

更偏体验时，你会更关注：

- TTFT
- streaming 体验
- 单请求延迟

更偏吞吐时，你会更关注：

- batch 效率
- 资源利用率
- 并发处理能力

### 2. 通用性 vs 极致性能

更通用的方案通常更容易上手、迁移和维护。  
更极致的方案通常需要更多硬件约束和工程成本。

### 3. 简单度 vs 系统深度

有些方案让你更快“先跑起来”，有些方案让你更容易继续向大规模场景扩。  
学习阶段通常先选前者，理解够了再接触后者会更顺。

## 对当前仓库最重要的结论

在这套学习型工程里，我们不追求一开始把真实 serving stack 全接满。  
更合理的路径是：

1. 先理解最小服务接口
2. 再理解 streaming / metrics / request tracing
3. 之后再把 mock 替换成更真实后端

所以当前仓库里的 `inference-service` 更像“学习入口”，不是最终选型结论。
