# Triton 与 TensorRT-LLM

## 为什么这两个名字也经常绑在一起

因为它们常出现在更偏 NVIDIA 生态、工程化部署和性能优化的讨论里。

但它们不是同一层东西：

- Triton 更像服务管理和模型部署入口
- TensorRT-LLM 更偏底层执行优化和性能路径

## Triton 更适合先理解成什么

先把 Triton 理解成“统一管理模型服务的一层外壳”。  
它的价值通常体现在：

- 模型如何组织和装载
- 服务如何统一暴露
- 多模型如何共存

所以它比较像“服务控制平面 + 执行入口”的组合。

## TensorRT-LLM 更适合先理解成什么

它更像“把模型在特定硬件上跑得更高效”的底层优化路线。  
也就是说，它离算子、图优化、硬件执行更近。

学习时最重要的不是把实现细节背下来，而是建立一个分层感：

- Triton：更靠服务接入和部署
- TensorRT-LLM：更靠执行效率和硬件优化

## 学习时最值得关注什么

不要把所有名字都当成“同一层框架互相替代”。  
更好的理解方式是问：

1. 这是服务层问题，还是执行层问题？
2. 它是在解决统一接入，还是在解决极致性能？

这会让你后面看选型时不容易混。
