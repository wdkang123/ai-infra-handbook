# 从学习型服务到真实 Serving Stack

## 为什么这一页值得补

因为当前仓库里的 `inference-service` 是故意做成学习型骨架的。  
这对入门非常好，但你后面一定会自然地问：

- 如果我要把它继续往真实系统推，下一步会长什么样？

这一页就是用来回答这个问题的。

## 当前学习型服务已经解决了什么

当前这层已经把最重要的 serving 结构讲清楚了：

- `/health`
- `/metrics`
- `/v1/chat/completions`
- 最小 streaming
- `x-request-id`

也就是说，它已经是一个“能让你看懂服务边界”的好入口。

## 它还故意没有解决什么

它还没有直接把这些重问题全部接满：

- 真实 vLLM / SGLang 执行后端
- tokenizer / chat template 处理
- 更复杂 batching / scheduling
- 真正的 prefix caching 指标
- 更完整的取消、超时、重试语义

这不是缺陷，而是学习节奏控制。

## 如果要往真实 stack 推，最自然的顺序是什么

更稳的顺序通常不是“一口气全接上”，而是分层推进：

### 第一步：替换执行后端

先把 mock executor 换成真实后端，比如 vLLM。

### 第二步：保住现有服务边界

也就是尽量保持：

- 接口不乱
- health / metrics 继续存在
- request id 继续保留

### 第三步：再补真实指标和运行时信息

比如更真实的 token 统计、cache 观察点、上游执行状态。

## 为什么这个顺序重要

因为它能帮助你保住一件很重要的事：

**结构先稳定，再让内部逐渐变真。**

如果一开始就把真实框架、复杂指标、运行时行为一起堆进来，  
学习成本会一下升得很高。

## 学习者最容易犯的错误

### “既然是学习项目，就应该立刻接真实框架”

不一定。  
更好的做法是先确认自己已经真的理解了边界，再去替换内部实现。

### “只要接上 vLLM，这层就算完成了”

也不够。  
真实 serving stack 还涉及：

- 观测
- 错误路径
- streaming 行为
- 和 gateway 的边界

### “学习型实现和真实系统是两套东西”

也不对。  
更准确的理解是：学习型实现是在为真实系统做结构预演。

## 这一章学完应该带走什么

当前 `inference-service` 不是假的，它是在用更低复杂度，把真实 serving stack 最重要的结构先讲清楚。  
后面你继续往真实框架推进时，最该保住的是这些结构，而不是急着把所有实现一次接满。
