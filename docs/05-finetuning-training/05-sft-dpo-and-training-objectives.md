# SFT、DPO 与训练目标

训练最怕的一种假进展是：

- 数据准备了
- 命令能跑
- checkpoint 生成了
- export 也有了

但最后没人能说清：

> 这次训练到底想让模型变好在哪？

所以学习微调时，不能只看“有没有开始跑”。
你必须先理解训练目标。

SFT 和 DPO 的重要性，不在于它们是两个热门缩写，而在于它们代表了两类不同的训练目标。

## 训练目标先于训练命令

训练命令回答的是：

> 怎么跑？

训练目标回答的是：

> 为什么跑，想让模型往哪个方向变？

如果目标没想清楚，训练很容易变成“把数据喂进去试试”。
这会带来几个问题：

- 数据格式不知道该怎么设计。
- loss 下降也不知道是否代表有用。
- eval 不知道该测什么。
- 训练后变差也不知道原因。
- 产物生成了但无法判断是否值得保留。

所以正确顺序应该是：

```text
任务问题 -> 训练目标 -> 数据结构 -> 训练方法 -> 评测方式 -> 发布判断
```

而不是从命令开始。

## SFT 解决什么问题

SFT 是 Supervised Fine-Tuning，可以先理解成：

> 用示范答案教模型更像目标输出。

它适合的场景是：你知道输入是什么，也知道希望模型输出什么。

例如：

- 给定用户问题，输出固定格式的回答。
- 给定客服记录，输出标准化工单摘要。
- 给定代码片段，输出符合团队风格的解释。
- 给定领域问答，输出更符合示例语气的答案。

SFT 的数据通常像这样：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个严谨的学习助手。"},
    {"role": "user", "content": "什么是 KV Cache？"},
    {"role": "assistant", "content": "KV Cache 是推理阶段保存已计算 key/value 状态的缓存..."}
  ]
}
```

这类数据表达的是：遇到类似输入时，请更接近这个目标输出。

## SFT 的优势和边界

SFT 的优势是直观、容易开始、数据结构清楚。
它非常适合学习微调第一站。

但 SFT 也有边界：

| 问题 | 说明 |
| --- | --- |
| 容易学到表面风格 | 数据质量差时，模型可能只模仿语气 |
| 不直接表达偏好比较 | 它告诉模型目标答案，不告诉模型两个答案谁更好 |
| 对覆盖范围敏感 | 数据没覆盖的情况，泛化不一定稳定 |
| 可能固化错误 | 示例答案错误时，模型会学习错误 |

所以 SFT 不是“把数据放进去就会变好”。
它要求你认真设计示范数据和评测方式。

## DPO 解决什么问题

DPO 是 Direct Preference Optimization，可以先理解成：

> 让模型从偏好比较里学习更应该选择哪类回答。

它关注的不是“这里有一个标准答案，请模仿”，而是：

- 对同一个输入，回答 A 比回答 B 更好。
- 模型应该更偏向 chosen，远离 rejected。

DPO 数据通常更像：

```json
{
  "prompt": "解释 TTFT 和 ITL 的区别",
  "chosen": "TTFT 是首 token 等待时间，ITL 是相邻 token 间隔...",
  "rejected": "TTFT 和 ITL 都是速度指标，差不多..."
}
```

这里训练信号来自比较关系，而不是单个标准答案。

## DPO 和 SFT 的核心差异

| 维度 | SFT | DPO |
| --- | --- | --- |
| 目标 | 模仿示范答案 | 学习偏好排序 |
| 数据 | 输入 + 目标输出 | 输入 + chosen + rejected |
| 适合阶段 | 建立基础任务行为 | 优化回答偏好与对齐 |
| 评测重点 | 是否符合示例风格和格式 | 是否更偏向高质量回答 |
| 常见风险 | 数据覆盖不足、学错风格 | 偏好数据噪声、比较标准不清 |

这不是谁更高级的问题。
它们服务于不同目标，也经常出现在不同训练阶段。

## SFT / DPO 怎么选

可以先用这张表判断自己真正要解决的问题：

| 你观察到的问题 | 更可能需要 | 原因 |
| --- | --- | --- |
| 模型不知道应该按什么格式回答 | SFT | 需要大量示范输出建立任务行为 |
| 模型回答风格太短、太散、缺少结构 | SFT 或 SFT 后 DPO | 先让它见过高质量结构，再用偏好强化选择 |
| 模型能给出两种回答，但经常选不好的那种 | DPO | 问题更像偏好排序，而不是不会生成 |
| 模型缺少领域知识 | 先做检索或数据增强，再谨慎 SFT | 微调不是知识库同步工具 |
| 模型在安全、礼貌、拒答边界上不稳定 | DPO 或其他对齐方法 | 需要清楚的 chosen/rejected 标准 |
| 模型输出 JSON 经常破格式 | SFT + 强格式 eval | 先示范格式，再用评测拦截退化 |

真实项目里经常不是二选一。
更稳的顺序通常是：先用 SFT 建立任务行为，再用偏好数据修正“会做但不总是选好”的部分。

## 为什么数据结构会跟着目标变

训练目标决定数据结构。

如果目标是“让模型按示例输出”，你需要 SFT 风格数据。
如果目标是“让模型更偏向某类回答”，你需要 preference 数据。
如果目标是“让模型学会工具调用”，你可能还需要工具轨迹数据。
如果目标是“减少特定错误”，你需要覆盖错误模式和反例。

所以设计数据前，先问：

- 我想改变模型的知识、格式、语气、推理过程，还是偏好？
- 我能不能给出足够高质量的目标输出？
- 我能不能给出稳定的偏好比较？
- 评测能不能反映这个目标？

这些问题比先调 LoRA rank 更重要。

## 数据质量检查清单

训练数据不是越多越好。
在开始训练前，至少要检查：

| 检查项 | SFT 数据 | DPO 数据 |
| --- | --- | --- |
| 输入是否真实 | prompt 是否来自真实任务或合理模拟 | prompt 是否覆盖真实比较场景 |
| 输出是否可示范 | assistant answer 是否值得模型模仿 | chosen 是否明显优于 rejected |
| 标准是否一致 | 格式、语气、长度、术语是否稳定 | 偏好规则是否一致，不是标注者心情 |
| 错误是否可追踪 | 是否知道哪些样本可能有误 | rejected 是否代表具体错误类型 |
| 覆盖是否平衡 | 是否覆盖短问、长问、边界输入 | 是否覆盖不同失败模式 |
| 脱敏是否完成 | 是否移除了个人信息和私有数据 | chosen/rejected 中是否有敏感内容 |

如果数据质量不稳，训练会把噪声压进模型里。
这时 loss 下降反而危险，因为它可能说明模型正在更努力地学习错误模式。

## 为什么评测方式也会跟着目标变

SFT 后你可能更关心：

- 输出格式是否稳定
- 是否遵守 system prompt
- 是否覆盖领域术语
- 是否减少无关内容
- 是否保留基础能力

DPO 后你可能更关心：

- chosen 类回答是否更常出现
- rejected 类错误是否减少
- 安全性或礼貌性是否提升
- 模型是否过度迎合偏好
- 原有任务能力是否退化

如果训练目标不同，却用同一套 eval 盲测，很容易得出错误结论。

## 把训练目标写进 eval

训练目标应该能翻译成评测项。
否则你无法判断训练是否真的有价值。

| 训练目标 | 对应 eval 应该看什么 |
| --- | --- |
| 输出更像教程 | 是否包含场景、机制、取舍、误区、代码入口 |
| 输出更稳定的 JSON | JSON parse 成功率、schema 覆盖、错误字段分布 |
| 术语解释更准确 | 关键概念命中率、幻觉率、引用是否合理 |
| 更偏向高质量回答 | chosen win rate、judge reason、人工抽样复核 |
| 减少某类错误 | 错误样本回归集、失败类型计数 |
| 保留基础能力 | 通用任务 smoke、旧数据集回归、长短输入对比 |

在当前仓库里，可以把这个思想映射到 `eval-module`：

- run 负责产生评测事实。
- sample outputs 负责保留样本级证据。
- compare 负责判断 candidate 是否相对 baseline 有变化。
- release recommendation 负责把变化转成 `approve / review / block` 建议。

也就是说，训练不是以 checkpoint 结束，而是要进入 eval 和发布门禁。

## 当前仓库为什么先做 SFT/LoRA 骨架

当前 `finetune-demo` 是学习型项目。
它还不是完整训练平台，也没有真实实现 DPO。

它优先表达的是训练工程骨架：

- config
- dataset validation
- dataset summary
- dataset registry
- run manifest
- checkpoint index
- export manifest
- run history
- export history

这是因为无论你未来做 SFT、DPO、RLHF，还是别的训练目标，这些工程对象都很重要。

先把训练资产链路学清楚，再接真实 SFT/DPO 栈会更稳。

## 一个具体判断场景

假设你的模型回答 AI Infra 概念时经常太短、太像提纲。
你有两种训练思路。

### SFT 思路

准备一批高质量教程式回答，让模型模仿：

- 有场景
- 有机制解释
- 有表格
- 有常见误区
- 有仓库映射
- 有自测问题

这适合解决“输出风格和结构不稳定”。

### DPO 思路

为同一个问题准备两种回答：

- chosen：深入、具体、有例子
- rejected：空泛、只有要点

让模型学会更偏向 chosen。

这适合解决“模型知道两种写法，但需要更偏好高质量写法”。

两个思路都可能有价值，但数据形态和评测方式不同。

## 常见失败模式与处理

训练目标不清楚时，失败通常会以这些形态出现：

| 失败模式 | 可能原因 | 处理方式 |
| --- | --- | --- |
| loss 下降但回答没变好 | eval 没对齐目标，或数据只是重复已有能力 | 回到目标，重新设计评测样本 |
| 回答变得更像模板 | SFT 数据风格过窄 | 增加多样化示例，避免只学固定套话 |
| 某些能力退化 | 数据覆盖过窄或训练过强 | 加入回归评测，降低训练强度或调整数据 |
| DPO 后过度迎合 | chosen/rejected 标准偏向讨好 | 重新定义偏好规则，加入安全和真实性约束 |
| JSON 格式仍然不稳 | 只训练了样子，没有强验证 | 增加 schema eval、失败样本和修复回归 |
| checkpoint 很多但没人敢用 | 没有 run manifest、eval compare 和发布记录 | 补齐资产链路，把训练产物接入证据库 |

这些问题说明：微调工程的核心不是“跑出一个 adapter”，而是让数据、训练、评测、发布判断形成闭环。

## 常见误区

### “SFT 是基础，DPO 一定更高级”

不一定。
DPO 不是 SFT 的简单升级版，它解决的是偏好优化问题。

### “只要 loss 下降，训练目标就达成了”

不一定。
loss 是训练过程指标，不等于真实任务质量。

### “有数据就能训练”

不够。
你需要知道数据表达的训练目标是什么。

### “DPO 不需要好数据”

不对。
偏好数据质量很关键，chosen/rejected 标准不稳定会把模型带偏。

### “当前仓库没真实 DPO，所以不用讲”

要讲。
因为学习者需要先理解训练目标差异，才知道当前 demo 的边界在哪里。
