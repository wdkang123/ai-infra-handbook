# SFT、DPO 与训练目标

训练最怕的一种假进展是：

- 数据准备了
- 命令能跑
- checkpoint 生成了
- export 也有了

但最后没人能说清：

> 这次训练到底想让模型变好在哪？

所以学习微调时，不能只看“有没有开始跑”。
你必须先理解训练目标。

SFT 和 DPO 的重要性，不在于它们是两个热门缩写，而在于它们代表了两类不同的训练目标。

## 训练目标先于训练命令

训练命令回答的是：

> 怎么跑？

训练目标回答的是：

> 为什么跑，想让模型往哪个方向变？

如果目标没想清楚，训练很容易变成“把数据喂进去试试”。
这会带来几个问题：

- 数据格式不知道该怎么设计。
- loss 下降也不知道是否代表有用。
- eval 不知道该测什么。
- 训练后变差也不知道原因。
- 产物生成了但无法判断是否值得保留。

所以正确顺序应该是：

```text
任务问题 -> 训练目标 -> 数据结构 -> 训练方法 -> 评测方式 -> 发布判断
```

而不是从命令开始。

## SFT 解决什么问题

SFT 是 Supervised Fine-Tuning，可以先理解成：

> 用示范答案教模型更像目标输出。

它适合的场景是：你知道输入是什么，也知道希望模型输出什么。

例如：

- 给定用户问题，输出固定格式的回答。
- 给定客服记录，输出标准化工单摘要。
- 给定代码片段，输出符合团队风格的解释。
- 给定领域问答，输出更符合示例语气的答案。

SFT 的数据通常像这样：

```json
{
  "messages": [
    {"role": "system", "content": "你是一个严谨的学习助手。"},
    {"role": "user", "content": "什么是 KV Cache？"},
    {"role": "assistant", "content": "KV Cache 是推理阶段保存已计算 key/value 状态的缓存..."}
  ]
}
```

这类数据表达的是：遇到类似输入时，请更接近这个目标输出。

## SFT 的优势和边界

SFT 的优势是直观、容易开始、数据结构清楚。
它非常适合学习微调第一站。

但 SFT 也有边界：

| 问题 | 说明 |
| --- | --- |
| 容易学到表面风格 | 数据质量差时，模型可能只模仿语气 |
| 不直接表达偏好比较 | 它告诉模型目标答案，不告诉模型两个答案谁更好 |
| 对覆盖范围敏感 | 数据没覆盖的情况，泛化不一定稳定 |
| 可能固化错误 | 示例答案错误时，模型会学习错误 |

所以 SFT 不是“把数据放进去就会变好”。
它要求你认真设计示范数据和评测方式。

## DPO 解决什么问题

DPO 是 Direct Preference Optimization，可以先理解成：

> 让模型从偏好比较里学习更应该选择哪类回答。

它关注的不是“这里有一个标准答案，请模仿”，而是：

- 对同一个输入，回答 A 比回答 B 更好。
- 模型应该更偏向 chosen，远离 rejected。

DPO 数据通常更像：

```json
{
  "prompt": "解释 TTFT 和 ITL 的区别",
  "chosen": "TTFT 是首 token 等待时间，ITL 是相邻 token 间隔...",
  "rejected": "TTFT 和 ITL 都是速度指标，差不多..."
}
```

这里训练信号来自比较关系，而不是单个标准答案。

## DPO 和 SFT 的核心差异

| 维度 | SFT | DPO |
| --- | --- | --- |
| 目标 | 模仿示范答案 | 学习偏好排序 |
| 数据 | 输入 + 目标输出 | 输入 + chosen + rejected |
| 适合阶段 | 建立基础任务行为 | 优化回答偏好与对齐 |
| 评测重点 | 是否符合示例风格和格式 | 是否更偏向高质量回答 |
| 常见风险 | 数据覆盖不足、学错风格 | 偏好数据噪声、比较标准不清 |

这不是谁更高级的问题。
它们服务于不同目标，也经常出现在不同训练阶段。

## 为什么数据结构会跟着目标变

训练目标决定数据结构。

如果目标是“让模型按示例输出”，你需要 SFT 风格数据。
如果目标是“让模型更偏向某类回答”，你需要 preference 数据。
如果目标是“让模型学会工具调用”，你可能还需要工具轨迹数据。
如果目标是“减少特定错误”，你需要覆盖错误模式和反例。

所以设计数据前，先问：

- 我想改变模型的知识、格式、语气、推理过程，还是偏好？
- 我能不能给出足够高质量的目标输出？
- 我能不能给出稳定的偏好比较？
- 评测能不能反映这个目标？

这些问题比先调 LoRA rank 更重要。

## 为什么评测方式也会跟着目标变

SFT 后你可能更关心：

- 输出格式是否稳定
- 是否遵守 system prompt
- 是否覆盖领域术语
- 是否减少无关内容
- 是否保留基础能力

DPO 后你可能更关心：

- chosen 类回答是否更常出现
- rejected 类错误是否减少
- 安全性或礼貌性是否提升
- 模型是否过度迎合偏好
- 原有任务能力是否退化

如果训练目标不同，却用同一套 eval 盲测，很容易得出错误结论。

## 当前仓库为什么先做 SFT/LoRA 骨架

当前 `finetune-demo` 是学习型项目。
它还不是完整训练平台，也没有真实实现 DPO。

它优先表达的是训练工程骨架：

- config
- dataset validation
- dataset summary
- dataset registry
- run manifest
- checkpoint index
- export manifest
- run history
- export history

这是因为无论你未来做 SFT、DPO、RLHF，还是别的训练目标，这些工程对象都很重要。

先把训练资产链路学清楚，再接真实 SFT/DPO 栈会更稳。

## 一个具体判断场景

假设你的模型回答 AI Infra 概念时经常太短、太像提纲。
你有两种训练思路。

### SFT 思路

准备一批高质量教程式回答，让模型模仿：

- 有场景
- 有机制解释
- 有表格
- 有常见误区
- 有仓库映射
- 有自测问题

这适合解决“输出风格和结构不稳定”。

### DPO 思路

为同一个问题准备两种回答：

- chosen：深入、具体、有例子
- rejected：空泛、只有要点

让模型学会更偏向 chosen。

这适合解决“模型知道两种写法，但需要更偏好高质量写法”。

两个思路都可能有价值，但数据形态和评测方式不同。

## 常见误区

### “SFT 是基础，DPO 一定更高级”

不一定。
DPO 不是 SFT 的简单升级版，它解决的是偏好优化问题。

### “只要 loss 下降，训练目标就达成了”

不一定。
loss 是训练过程指标，不等于真实任务质量。

### “有数据就能训练”

不够。
你需要知道数据表达的训练目标是什么。

### “DPO 不需要好数据”

不对。
偏好数据质量很关键，chosen/rejected 标准不稳定会把模型带偏。

### “当前仓库没真实 DPO，所以不用讲”

要讲。
因为学习者需要先理解训练目标差异，才知道当前 demo 的边界在哪里。
