# SFT、DPO 与训练目标

## 为什么训练不能只看“有没有开始跑”

因为训练不是为了把 GPU 跑满，而是为了把模型往某个目标方向推。

如果目标本身没想清楚，你就会很容易陷入一种假进展：

- 数据准备了
- 命令能跑
- 产物也生成了

但最后很难回答：  
**这次训练到底想让模型变好在哪。**

## SFT 更适合先理解成什么

SFT 可以先理解成“监督式把模型往目标答案风格上拉近”。

它最适合帮助你建立训练第一直觉：

- 我给模型示范输入输出
- 我希望模型更像这些示范

所以它通常是学习微调时最自然的第一站。

## DPO 更适合先理解成什么

DPO 更适合先理解成“不是只教模型一个标准答案，而是让模型在偏好比较里学会更偏向哪种回答”。

这会把你的训练目标从“模仿答案”推进到“学习偏好”。

所以 SFT 和 DPO 的差异，不只是名字不同，而是它们对应的训练目标不同：

- SFT 更像直接示范
- DPO 更像偏好对齐

## 为什么这个区分对学习很重要

因为很多初学者会把所有训练都理解成“喂数据让模型学”。  
但一旦你开始区分训练目标，就会发现：

- 数据格式会不同
- 损失形式会不同
- 你关心的评测方式也会不同

这会直接影响后面你怎么设计数据、怎么解释结果、怎么判断一次 run 有没有意义。

## 在当前仓库里为什么还没把 DPO 做成真实实现

因为当前 `finetune-demo` 的目标是学习骨架，不是完整训练平台。  
我们优先保留的是：

- run
- artifacts
- checkpoint
- export
- history

这样你能先把训练工程对象学明白，再逐步决定什么时候接更真实的 SFT、DPO 或 RLHF 栈。

这也和整个项目原则一致：  
先把结构讲清楚，再在你后续逐步实操时细化实现。

## 学习时最值得先建立的三个判断

### 判断一：训练目标先于训练命令

先知道自己是在做 SFT、偏好优化，还是别的目标，再去看命令和配置。

### 判断二：数据结构跟着目标走

SFT 的数据更像输入输出示范；  
DPO 的数据更像偏好对。

### 判断三：评测方式也会跟着目标变

如果你的训练目标不同，后面看结果时最关心的指标和评测方式也会不同。

## 这一章学完应该带走什么

训练不是一套统一按钮，而是围绕不同训练目标组织出来的流程。  
把 SFT、DPO 这种目标层差异看清楚之后，你后面再看数据集、run、checkpoint、export，理解会稳很多。
