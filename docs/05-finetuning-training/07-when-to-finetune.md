# 什么时候该微调

“能微调”不等于“应该微调”。

这是学习 LLM 工程时非常重要的一条判断。
很多人一看到 LoRA、QLoRA、Unsloth、checkpoint、export，就会自然觉得：既然工具已经有了，那当然要尽快训练。

但更成熟的工程判断是先问：

> 当前问题真的是模型参数问题吗？

如果问题其实来自 prompt、workflow、RAG、工具调用、数据源、评测方式或产品边界，微调可能会花很多成本，却没有解决真正的问题。

## 一句话判断

微调更适合解决“稳定行为模式”的问题，不适合解决“动态知识来源”的问题。

可以先用这句话自查：

```text
如果明天知识变了，我希望模型立刻更新吗？
如果希望，优先 RAG 或工具；
如果不是，而是希望模型稳定学会一种格式、风格、偏好或领域回答模式，再考虑微调。
```

这不是绝对规则，但足够挡住很多过早训练。

## 先给一个判断框架

可以把改进 LLM 应用的手段粗略分成几层：

| 手段 | 更适合解决 | 成本与风险 |
| --- | --- | --- |
| Prompt / system instruction | 输出格式、语气、少量行为约束 | 成本低，迭代快，但稳定性有限 |
| Workflow / agent flow | 多步骤任务、工具顺序、业务流程 | 需要系统设计，适合流程问题 |
| RAG / retrieval | 缺知识、缺最新信息、需要引用资料 | 依赖数据质量和检索质量 |
| Tool use | 需要计算、查询、执行外部动作 | 需要工具接口和错误处理 |
| Eval / guardrail | 判断好坏、拦截明显风险 | 不能直接让模型变强，但能治理质量 |
| Fine-tuning | 稳定行为、格式、风格、偏好或领域模式 | 成本更高，需要数据、训练和回归 |

微调不是最高级的唯一答案。
它是其中一种工具，适合特定问题。

## 不该先微调的情况

### 1. 任务边界还没稳定

如果你还在频繁修改：

- 用户输入是什么
- 输出格式是什么
- 系统应该调用哪些工具
- 是否需要检索
- 成功标准是什么

这时太早微调，容易把不稳定需求固化进模型。

更好的做法是先用 prompt、workflow 和小规模 eval 把任务边界跑清楚。

### 2. 缺的是知识，不是行为

如果模型主要问题是“不知道你的私有资料、产品文档、最新政策”，优先考虑 RAG。

微调并不适合把大量变化频繁的事实知识硬塞进参数里。
知识更新快、需要引用来源、需要可撤回时，检索通常更合适。

### 3. 没有稳定评测

如果你还不能回答：

- 训练后怎么判断变好了？
- 哪些样本不能退化？
- 失败样本怎么分析？
- 分数上涨多少才值得保留？

那微调很容易变成盲试。

在当前项目里，训练产物应该接到 eval-module 的 run/compare/history，再形成发布判断。

### 4. 数据质量还很差

如果训练数据里有：

- 错误答案
- 重复样本
- 格式混乱
- 角色缺失
- 任务目标不一致
- 低质量合成数据

模型可能会认真学习这些问题。
微调不会自动把坏数据变好。

### 5. 问题可以通过后处理解决

如果只是输出 JSON 字段名不稳定，可能用 structured output、schema 校验、retry 或后处理就够了。
不一定要微调。

## 可以考虑微调的情况

### 1. 你需要稳定的输出格式和行为

例如：

- 总是按某种结构写分析
- 总是输出特定字段
- 总是遵循领域回答风格
- 总是在回答里保留推理边界

如果 prompt 能做到一部分，但在不同输入下不稳定，SFT 可能有价值。

### 2. 你有高质量示范数据

微调最怕“数据只是看起来很多”。
更有价值的是少量但高质量、目标一致、格式稳定的数据。

你应该能说清：

- 数据来自哪里
- 为什么这些答案是好答案
- 覆盖了哪些任务模式
- 哪些错误模式没有覆盖
- 是否有数据版本和 sha256

### 3. 你想优化偏好

如果问题不是“不会回答”，而是“经常在两种回答风格里选错”，DPO 或偏好优化可能更合适。

例如：

- 更偏向具体而不是空泛
- 更偏向带证据而不是泛泛总结
- 更偏向遵守格式而不是自由发挥

这时你需要的是 chosen/rejected 偏好数据。

### 4. Prompt 已经很重

有些系统为了让模型稳定，prompt 越写越长。
如果大量 prompt 只是为了反复说明固定行为，微调可能把一部分行为内化，减少 prompt 负担。

但这也要谨慎：不能为了省 prompt 就牺牲可控性和可更新性。

### 5. 你已经有回归路径

微调前最好已经有：

- baseline run
- candidate run
- compare report
- regression cases
- release gate
- history

这样训练结果才能进入工程闭环，而不是只看“感觉更好”。

## 一条实用决策路径

可以按这个顺序问：

1. 这个问题能不能通过 prompt 改好？
2. 如果不行，是不是 workflow 或工具边界没设计好？
3. 如果不行，是不是缺知识，需要 RAG？
4. 如果不行，有没有稳定 eval 判断好坏？
5. 如果有，是否有高质量训练数据？
6. 如果有，训练目标是 SFT、DPO，还是别的？
7. 训练后如何 compare、回归和发布？

只有走到后几步，微调才更像正确工具。

## 决策矩阵

| 症状 | 优先手段 | 微调是否合适 |
| --- | --- | --- |
| 模型不知道最新文档 | RAG | 通常不优先 |
| 输出 JSON 偶尔不合法 | schema/retry/structured output | 先不急 |
| 回答总是太空泛 | prompt/workflow/eval | 有高质量样本后可考虑 |
| 长期需要固定风格 | SFT | 可能合适 |
| 两种回答都对，但偏好不同 | DPO / preference tuning | 可能合适 |
| 工具调用顺序混乱 | workflow/tool orchestration | 不一定 |
| 某领域术语表达稳定错误 | 数据 + SFT + eval | 可考虑 |

矩阵的重点是：先识别问题类型，再选工具。

## 一个具体场景：教程内容太薄

假设你希望模型把 AI Infra 章节写得更深入。
现在输出经常像提纲，没有场景、机制和误区。

你可以先试：

### Prompt

在提示词里明确要求：

- 写真实场景
- 解释机制
- 给表格
- 写仓库映射
- 写常见误区
- 写自测问题

如果质量稳定，先不微调。

### Workflow

把生成拆成步骤：

1. 先列读者问题。
2. 再列工程场景。
3. 再扩机制。
4. 再补仓库映射。
5. 最后自查是否太提纲。

如果流程能稳定改善，也先不微调。

### Eval

建立“深度教程页标准”，检查页面是否覆盖关键要素。
没有 eval，不建议训练。

### Fine-tuning

当你已经积累了足够多高质量章节样本，并且希望模型稳定学习这种写法，SFT 或 DPO 才更有意义。

这个例子很贴近当前项目：我们现在正在先把内容标准和样本做扎实，而不是急着训练一个写作模型。

## 微调前的最低门槛

如果真的准备训练，至少先满足：

```text
[ ] 任务边界稳定
[ ] 有 baseline
[ ] 有 golden set
[ ] 有高质量训练样本
[ ] 数据有 version 和 sha
[ ] 训练参数能进入 manifest
[ ] checkpoint 能进入 index
[ ] export 能追溯 source checkpoint
[ ] eval 能比较 baseline/candidate
[ ] 有人工复核失败样本的流程
```

这些门槛不是为了增加流程，而是为了避免训练结束后只剩一个“看起来不错”的 adapter。

## 当前仓库怎么帮助你判断

`finetune-demo` 不是在鼓励“一上来就训练”。
它是在帮你学会：如果决定训练，至少要把工程对象留清楚。

相关能力包括：

- dataset validation
- dataset summary
- dataset registry
- run manifest
- checkpoint index
- export manifest
- run history
- export history

同时，`eval-module` 提供：

- run
- compare
- leaderboard
- list-runs
- list-comparisons

两者合在一起，表达的是：

```text
决定是否训练 -> 训练 -> 产物追踪 -> 评测 -> 发布判断
```

微调不能孤立存在。
它必须接回评测和发布流程。

## 微调后的发布判断

训练完成后，建议按这个顺序判断：

1. export 是否成功，manifest 是否完整。
2. candidate eval 是否跑通。
3. baseline/candidate 是否同 task、同样本、同 judge。
4. compare 是否 improved，而不是只看单次分数。
5. 失败样本是否可接受。
6. serving 侧格式、延迟、成本是否可接受。
7. 是否需要继续训练、回滚，还是只把样本加入下一轮数据。

如果 eval 没赢，训练不一定没价值；它可能暴露了数据问题、目标问题或评测口径问题。

## 微调和 RAG 不是二选一

真实系统经常组合使用：

- RAG 负责引入可更新知识。
- Prompt 负责当前任务约束。
- Tools 负责外部动作。
- Fine-tuning 负责稳定风格、格式或偏好。
- Eval 负责持续判断质量。

所以不要把问题问成：

> 我应该做 RAG 还是微调？

更好的问题是：

> 哪部分问题应该由 RAG 解决，哪部分问题才值得交给微调？

## 常见误区

### “微调总比 prompt engineering 高级”

不一定。
如果 prompt 能稳定解决，微调可能只是增加复杂度。

### “知识问题就靠微调”

不稳。
频繁变化、需要来源、需要撤回的知识，更适合 RAG。

### “小实验不用 eval”

风险很大。
越是小实验，越容易被个别样本的体感误导。

### “训练数据越多越好”

不一定。
低质量数据越多，可能越稳定地学坏。

### “微调后一定能省成本”

不一定。
训练、维护、评测、回归和部署都有成本。是否省成本要算整体系统。

2. 什么时候 SFT 或 DPO 更有意义？

4. RAG、prompt、workflow、tool use 和 fine-tuning 如何分工？
