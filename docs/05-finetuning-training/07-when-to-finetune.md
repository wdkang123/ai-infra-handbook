# 什么时候该微调

## 为什么这是一道必须单独想清楚的问题

因为“能微调”不等于“应该微调”。

很多学习者一接触训练，就会自然觉得：  
既然有 LoRA、QLoRA、Unsloth、checkpoint、export，这条路当然就应该尽快走。

但更成熟的判断应该是先问：

- 当前问题真的是模型参数问题吗
- 还是提示词、工作流、检索、工具调用的问题

如果这个问题没想清楚，后面很容易花很多时间在训练上，却没有得到最值得的收益。

## 什么时候更像不该先微调

以下这些情况，往往更适合先不微调：

### 1. 你还没把 prompt 和 workflow 试清楚

如果连任务边界都还在快速变化，太早微调通常会把问题提前固化。

### 2. 你缺的是知识接入，不是行为风格

如果模型主要是“不了解你的数据或事实”，那先想检索、RAG、知识接入，往往比训练更直接。

### 3. 你还没有稳定评测方式

如果连“怎么判断训练后变好没变好”都还不稳，微调很容易变成盲试。

## 什么时候更像可以考虑微调

下面这些情况，微调会更有意义：

### 1. 你希望模型稳定呈现某种输出风格

比如更一致的结构、语气、格式、任务行为。

### 2. 你已经有了比较稳定的数据形式

这时训练才更像“有方向地推动模型”，而不是随机试运气。

### 3. 你已经有了最小评测与回归路径

也就是训练后你知道该怎么比、怎么看、怎么决定要不要保留结果。

## 为什么这件事和当前仓库有关

因为当前仓库里的 `finetune-demo` 并不是在劝你“一上来就训练”，  
而是在帮你把训练这条路的最小工程对象先讲清楚。

这会让你在真正决定要不要微调时，不再只是凭感觉，而是能问：

- 我有没有数据摘要
- 我有没有 run / history
- 我有没有 compare 和回归思路
- 我有没有明确训练目标

## 微调和别的路径不是非此即彼

这点也很重要。  
更现实的系统里，常常不是“只靠微调”或“只靠 RAG”，而是多种手段组合。

你可以把选择顺序想成：

1. 先把 prompt / workflow 跑顺
2. 再补 retrieval / tools
3. 再判断是否需要微调来稳定行为或风格

这样会比一开始就把训练当银弹更稳。

## 学习时常见误区

### “有训练工具就应该马上训练”

不一定。  
工具存在，不等于它是当前最优路径。

### “微调总比 prompt engineering 更高级”

也不对。  
很多时候 prompt 和 workflow 改对了，收益更快。

### “小规模训练不用评测也没关系”

风险仍然很大。  
哪怕是最小实验，也应该有最小回归方式。

## 这一章学完应该带走什么

微调是很重要的一条路，但它更适合在问题边界、数据形式和评测方法都稍微稳定之后再进入。  
真正成熟的学习路径，不是“尽快训练”，而是“知道什么时候训练最值得”。
