# Unsloth 与训练栈

## 为什么要单独讲 Unsloth

因为它很容易被误解。

很多人一听到 Unsloth，就会把它想成：

- 一个完整训练平台
- 一个和 PEFT/TRL 平级竞争的大框架
- 甚至是“训练版 vLLM”

这些理解都不太准。  
更适合的定位是：**它是微调栈里的加速层。**

## 先把训练栈的层次拆开

学习微调时，最好先把这几层分开看：

### 1. PyTorch

最底层计算和训练执行基础。

### 2. PEFT

更像统一接口层。  
LoRA、QLoRA 这些参数高效微调方法，通常会在这层被更稳定地组织起来。

### 3. TRL 或 Trainer 层

这层更像训练流程组织者。  
比如 SFT、DPO 这种训练任务，往往会在这里被包装成更可用的流程。

### 4. Unsloth

它更像“让前面这些训练更快、更省资源”的加速层。  
也就是说，它不是要替代整条训练栈，而是要让这条栈跑得更轻。

## 为什么它对学习者有吸引力

因为微调最容易卡在两个现实问题上：

- 显存不够
- 训练太慢

Unsloth 之所以常被提起，就是因为它正好对着这两个痛点。  
对学习者来说，这意味着：

- 更容易在有限硬件上做实验
- 更容易把注意力放在训练结构，而不是全被资源门槛挡住

## 学习时最该抓住的判断

### 判断一：Unsloth 不是新的训练范式

它不是在发明全新的微调理论。  
它更多是在让已有范式，尤其是 LoRA/QLoRA 这种路线，变得更容易实际运行。

### 判断二：它和 PEFT 更像协作关系

你不需要把它们理解成“谁取代谁”。  
更实用的理解是：

- PEFT 给你统一接口
- Unsloth 给你更轻的执行体验

### 判断三：它主要影响训练，不是推理

这点很重要。  
如果你把 Unsloth 误当成推理优化工具，后面就会把训练栈和 serving 栈搅在一起。

## 在当前仓库里怎么对应

当前 `finetune-demo` 不是完整接入 Unsloth 的训练系统，而是先把你最需要的训练工程骨架留出来：

- train
- artifacts
- checkpoint
- export
- history

所以在这套仓库里，Unsloth 更适合作为“你未来可替换接入的训练加速后端”来理解，而不是现在就要被做成高度真实实现。

这也符合我们整个项目的原则：  
先搭出适合学习的结构，再在你真正开始逐步实践时细化代码。

## 最适合的学习顺序

1. 先理解 LoRA / QLoRA / PEFT 这些概念边界
2. 再理解训练 run 应该沉淀哪些产物
3. 最后再看 Unsloth 为什么能让这条流程更轻

这个顺序会比一上来就追求“最快配置”更稳。

## 这一章学完后应该带走什么

最重要的不是记住 Unsloth 的所有特性，而是形成这个判断：

它属于训练栈里的加速层。  
你以后在真实硬件上做实验时，可以把它当成“让轻量微调更现实”的选项，而不是把它误当成完整训练平台。
