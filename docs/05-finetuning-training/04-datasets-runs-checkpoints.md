# 数据集、Run、Checkpoint

## 为什么训练学习不能只盯着模型和参数

因为一旦你真的开始跑实验，最容易失控的往往不是模型本身，而是“这次到底是怎么跑出来的”。

如果你没有把数据集、run、checkpoint 这些对象单独看清楚，后面很容易出现：

- 不知道这次训练到底用了哪份数据
- 不知道这个 checkpoint 是第几次实验留下的
- 不知道 export 对应的是哪次 run
- 两次结果差异很大，但找不到原因

所以训练工程的核心，不只是训练本身，还包括如何把训练过程组织成可回看的对象。

## 数据集为什么是第一层

因为它决定了“模型往哪个方向被推动”。

学习阶段你不用一开始就研究复杂数据工程，但至少要形成这个意识：

- 数据集不是训练前的前置杂务
- 它是实验语义的一部分

所以一个好的训练 run，至少应该留下数据集摘要，而不是只剩一个权重文件。

数据集摘要回答的是“这次 run 看到了什么”。  
dataset registry 更进一步，回答的是“这些训练输入在多次 run 之间如何被登记和追踪”。当前仓库用 `data/dataset_registry_entry.json` 和 `dataset_registry.jsonl` 表达这层关系。

如果你想从命令行查看 registry，可以用 `list-datasets` 生成 JSON/Markdown 报告。这个查询层不会改变训练产物，只是把已有 registry 读出来。

## Run 应该被理解成什么

run 最好被理解成“一次完整训练尝试的目录对象”。

它不只是一个命令执行结果，更应该至少包含：

- 训练配置
- 数据集摘要
- 指标
- 日志
- checkpoint 索引
- 最终产物说明

这样你后面看历史时，才能回答“这次到底做了什么”。

## Checkpoint 为什么不只是临时文件

很多初学者会把 checkpoint 当成训练过程里自动生成的中间垃圾。  
但从工程视角看，它很重要，因为它连接了：

- 恢复训练
- 对比不同阶段结果
- 导出最终产物
- 回溯某次实验状态

所以 checkpoint 更适合被理解成训练流程中的正式节点，而不是附属产物。

## Export 为什么一定要单独看

export 在很多项目里都被忽略，  
但其实它是在回答：

**“训练出来的东西，怎么变成后续可以继续使用的输出。”**

这一步如果不单独保留，后面：

- 部署
- 复用 adapter
- 对比多次实验

都会变得很混乱。

## 在当前仓库里怎么对应

当前 `finetune-demo` 已经把这些对象的学习骨架留出来了。  
你可以在项目里看到：

- run manifest
- artifacts manifest
- logs
- dataset summary
- dataset registry entry
- dataset registry report
- checkpoints
- export manifest
- history

这套结构的价值，不在于它已经是生产级训练平台，  
而在于它能让你先建立“训练工程资产到底长什么样”的直觉。

## 学习时常见误区

### “训练成功 = 生成了一个 checkpoint”

太窄了。  
更准确的说法应该是：  
训练成功，意味着一次 run 及其上下文被完整留下。

### “数据集以后再整理也行”

风险很大。  
因为一旦 run 跑出来，而数据来源和摘要没留下，后面就很难复现。

### “export 只是部署前的最后一步”

不只是。  
它也是训练工程里把实验结果变成可流转资产的关键步骤。

## 这一章学完应该带走什么

你最应该带走的是这个思路：

- 数据集定义实验方向
- run 组织一次训练尝试
- checkpoint 记录过程节点
- export 产出可使用资产

把这四层分开后，后面你再看更真实的训练平台、实验追踪系统或团队工作流时，就会顺很多。
