# 外部模型名与内部目标映射

## 为什么这个点值得单独讲

因为 gateway 最容易被低估的一件事，就是“名字不是小事”。

很多人第一次看平台层时，会默认认为：

- 用户传进来的模型名
- gateway 内部识别的模型名
- 下游真正执行的目标

应该天然是同一个东西。

但真实平台里，这三者往往不会完全一致。

当前 gateway 的 `/v1/models` 会把外部模型名、target model、fallback 数量和 upstream health 摆出来。它适合作为最小模型发现接口：调用方先知道平台允许请求哪些模型，再通过 chat completion 真正发起生成。

## 外部模型名在回答什么

外部模型名更像平台对调用方承诺的接口名。

它的价值是：

- 对调用方稳定
- 对应用层易理解
- 平台内部即使换实现，也不一定要强迫上游一起改

所以它更像 API 设计的一部分。

## 内部目标在回答什么

内部目标更像“gateway 最后到底把请求打给谁”。

它可能包含：

- 下游真实模型名
- 下游服务地址
- 某个 provider 的特殊标识
- 某个版本或某个 canary 目标

所以它的重点不是“好看”，而是“可执行”。

## 为什么映射层对平台很关键

因为有了这一层，平台才能同时拥有两种能力：

1. 对外保持稳定命名
2. 对内灵活切换执行目标

这正是 gateway 之所以不像“纯代理”的一个关键点。

## 映射层会带来什么能力

最直接的几个能力包括：

- 外部 API 稳定
- 版本切换更容易
- fallback 更容易组织
- canary 更容易挂接
- 多 provider / 多 backend 更容易统一

所以模型名映射其实不是“配置细节”，而是平台抽象本身的一部分。

## 在当前仓库里怎么对应

当前 `ai-gateway` 已经保留了这个最小结构：

- 外部请求带一个模型名
- gateway 识别这个模型名
- router 决定下游目标
- 再把请求打给 inference-service

也就是说，现在虽然还没有特别复杂的 provider 体系，但这条边界已经建立起来了。

## 学习时常见误区

### “模型名只是字符串，不值得专门设计”

其实非常值得。  
因为它会影响平台长期接口稳定性。

### “gateway 直接透传下游模型名就行”

短期也许方便，长期会让平台层抽象变薄，演进空间变小。

### “只有多模型平台才需要映射层”

哪怕当前只有一个模型，这层也值得先留出来。  
因为它先把“平台命名”和“执行命名”分开了。

## 这一章学完应该带走什么

外部模型名与内部目标的映射，不是小配置，而是平台层非常关键的一层抽象。  
它让“对外稳定接口”和“对内灵活执行”能同时成立。
