# Streaming、错误路径、Upstream Health

## 为什么平台层不只处理“成功请求”

很多人学 gateway 时，会先把注意力放在一条正常代理链路上：

- 请求进来
- 鉴权通过
- 路由成功
- 下游返回结果

但真正让平台层变得像平台的，往往不是成功路径，而是边界路径：

- `stream=true` 时会发生什么
- 下游挂掉时怎么表现
- 鉴权失败、模型不存在、限流命中时要不要统一返回结构
- `/health` 到底是在说“我活着”，还是“我的下游也大致正常”

这些问题会直接决定系统是不是可运维。

## Streaming 为什么特别值得单独看

因为一旦进入 streaming，请求就不再只是一个简单的“收包再回包”流程了。

平台层这时至少会面对：

- 上游事件流怎么继续往外转
- 中途出错时如何结束流
- request id 和日志怎么对齐
- metrics 要记完整请求，还是记事件流过程
- structured events 要记录 attempt、fallback、success 还是 stream error

所以 streaming 不是普通代理逻辑上加一个布尔值这么简单。  
它通常会把“入口层是不是只是透明代理”这个问题直接放大。

当前 `ai-gateway` 的处理原则是：如果首个 chunk 之前下游 `5xx` 失败，可以尝试 fallback；如果所有候选都失败，或者已经开始输出后才失败，就在 SSE 中发出结构化 `error` 事件并结束流。这样客户端至少能得到一个明确的失败对象，而不是只看到连接断开。

普通非流式请求还会用 `x-upstream-model` 和 `x-fallback-used` 表达最终命中的下游；`/metrics` 会记录 fallback attempts 和 successes；`/events` 会保留最近的 attempt、fallback、success 或 stream error。这样 fallback 不是静默发生，而是能被单次请求、最近事件和长期指标同时解释。

## 当前仓库里为什么要做最小 streaming 透传

因为这非常适合学习。

如果一开始就做复杂重试、背压、取消传播，你会很容易被实现细节淹没。  
而现在这套最小 streaming 透传，正好能让你先看清楚：

- inference-service 可以流式吐结果
- ai-gateway 可以继续把这条流转发出去
- 这件事和普通 JSON 响应在系统组织上是不同的

这就够你建立第一层直觉了。

## 错误路径为什么不是“以后再补”

因为平台层的职责之一，就是把失败变成可理解、可治理的系统行为。

学习阶段最值得先掌握的几类错误是：

- `401`：入口不合法
- `404`：模型或目标不存在
- `429`：入口层主动拒绝，保护下游
- `502`：上游失败，平台层无法拿到正常结果

这几类错误很有代表性，因为它们分别对应不同层次的问题：

- 身份
- 路由
- 治理
- 下游可用性

## Upstream Health 到底在表达什么

这是最容易被简化错的地方。

一个服务自己的 `/health` 可以很简单：  
“进程活着、配置加载了、路由注册了。”

但 gateway 的 `/health` 如果要更有意义，就不能只说“我自己活着”，  
它还应该尽量知道：

- 我依赖的下游是不是活着
- 如果下游死了，我是 `healthy`、`degraded` 还是 `unhealthy`

当前仓库里已经把这件事做到了最小真实探测：  
gateway 会探测 upstream health，并在失败时把顶层状态降成 `degraded`。

这一步很有学习价值，因为它让你看到：

- 平台层健康状态通常不是单机进程概念
- 它会天然带有依赖传播

## 学习时常见误区

### “streaming 只是响应格式不同”

不够准确。  
它通常会牵动代理方式、错误处理、日志、指标和排障方式。

### “错误路径只是补测试用的”

不对。  
平台层真正的价值之一，就是把失败清楚地表达出来，而不是让失败变成混乱。

### “health 只要 200 就行”

也不对。  
一个更成熟的学习视角是：

- 200 只是传输层面
- health payload 的结构和状态语义同样重要

## 这一章学完应该带走什么

平台层真正的成熟度，很多时候不是看它正常代理时多顺，而是看：

- streaming 时是否还能保持边界清晰

- 健康状态是否包含对下游依赖的最小真实判断
