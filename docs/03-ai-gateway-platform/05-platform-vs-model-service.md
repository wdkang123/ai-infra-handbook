# 平台层与模型服务层边界

## 为什么这条边界必须反复强调

因为 AI Infra 初学阶段最常见的混乱，就是把 gateway 和 inference-service 混成一个东西。

比如会下意识觉得：

- gateway 也该直接“懂模型”
- inference-service 也该顺手做鉴权和限流
- 只要服务能返回文本，边界细不细无所谓

但如果这条边界不清楚，后面你一旦要加：

- 多模型
- 多入口
- 评测
- 训练
- 观测

系统就会很快变乱。

## 平台层主要回答什么

平台层主要回答：

- 谁可以调用
- 这条请求该去哪里
- 入口层是否允许它通过
- 我怎么把跨服务请求串起来看

也就是说，它的视角更偏“系统治理”。

## 模型服务层主要回答什么

模型服务层主要回答：

- 这个模型调用如何被执行
- 普通响应和 streaming 如何产出
- metrics 如何暴露
- prompt 和 completion 的最小执行边界是什么

也就是说，它的视角更偏“推理执行”。

## 为什么两层分开后学习会更顺

因为你会逐渐形成一个非常稳定的判断框架：

### 涉及身份、入口、模型名映射、限流

先想 platform / gateway。

### 涉及执行、streaming、cache、生成结果

先想 inference / serving。

这个分层一旦建立，后面你再看更复杂系统，就不会总想着把所有逻辑塞进一个服务里。

## 当前仓库是怎么用代码把这件事讲清楚的

当前仓库其实就是故意用两个最小服务把这条边界拉出来：

### `inference-service`

更接近执行层。

### `ai-gateway`

更接近入口治理层。

这也是为什么我们一直没有把所有逻辑硬塞进一个应用里。  
对学习来说，先把边界看清楚，比一开始就堆满功能更重要。

## 学习时常见误区

### “模型服务顺手做鉴权也没问题”

当然可以做，但从长期结构看，这会让服务层和平台层职责混在一起。  
学习阶段最好先把它们分开理解。

### “gateway 只要会转发就够了”

不够。  
如果它只会转发，那它很难称得上平台层。

### “两层分开只是为了微服务好看”

也不对。  
分层的真正价值在于：

- 更清楚的职责边界
- 更容易演进
- 更容易观测与排障

## 你后面开始实操时，最值得问自己的问题

当你在文档和代码里看到某个功能点时，先问：

1. 这是入口治理问题，还是执行问题？
2. 如果把这一层换实现，另一层应不应该基本不动？

如果答案是“应该基本不动”，那说明边界划分大概率是对的。

## 这一章学完应该带走什么

AI Infra 很多复杂度，其实都来自边界不清。  
所以平台层与模型服务层的分离，不只是工程习惯，更是你后面继续学习所有内容的稳定坐标轴。
